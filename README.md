# 트렌드 TOP 10

관심 키워드의 시장현황(네이버 데이터랩)과, 검색어로 직접 조회하는 네이버 쇼핑 검색
결과 TOP10을 함께 보여주는 **개인 참고용** 대시보드입니다. **공식 순위 서비스가
아니라, 네이버 데이터를 화면용으로 정리해서 보여주는 개인 연습 프로젝트**라는 점을
꼭 기억해 주세요.

## 🔗 배포된 사이트

**https://trend-top10.onrender.com/**

브라우저에서 위 주소로 바로 접속해서 확인할 수 있습니다. Render 무료 플랜은 오랫동안
방문이 없으면 서버가 잠드는데, 오랜만에 접속하면 서버가 다시 깨어나는 동안 첫 로딩이
몇십 초 정도 느릴 수 있습니다. 정상 동작이니 잠시 기다렸다가 새로고침해 주세요.

> ⚠️ **API 키/Secret 주의사항**: 네이버 API Client ID/Secret 값은 GitHub 저장소나
> `.env` 파일에 커밋해서 올리면 안 됩니다. 반드시 Render의 "Environment Variables"
> 설정 화면에만 입력하세요. 자세한 방법은 아래 "Render 배포 설정값" 절을 참고하세요.

## 이 프로젝트는 무엇인가요?

"트렌드 TOP 10"은 화면을 **"1. 시장현황" → "2. 상품 검색" → "3. 상세 결과"** 순서로
살펴보는 개인 참고용 대시보드입니다.

1. **시장현황**: 미리 정해둔 관심 키워드 10개(선크림, 양산, 여름원피스, 쿨토시,
   제습기, 미니선풍기, 래쉬가드, 샌들, 냉감이불, 물놀이용품)의 최근 30일 검색·쇼핑
   클릭 추이를 네이버 데이터랩(검색어트렌드·쇼핑인사이트) API로 참고합니다.
   **네이버 공식 인기검색어 순위가 아니라, 설정한 관심 키워드 기준의 참고용
   시장현황**입니다.
2. **상품 검색**: 궁금한 검색어를 직접 입력하면 **네이버 쇼핑 검색 API**가 실제로
   돌려주는 검색 결과 TOP10(상품명·쇼핑몰·최저가·카테고리 등)을 조회합니다. 기본
   정렬은 정확도순(`sort=sim`)입니다.
3. **상세 결과**: 상품 검색 결과를 표/가격 요약 카드/카테고리 필터/가격 비교 차트로
   자세히 살펴보고, 결과 요약을 복사할 수 있습니다.

이 화면이 보여주는 모든 데이터는 **구매순위·판매순위·네이버 공식 인기순위가
아닙니다.** 코딩 초보자가 Claude Code와 함께 만들고 있는 연습 프로젝트이며, 빌드
도구나 외부 라이브러리 없이 `index.html` 파일 하나로 동작합니다. 네이버 API 응답
원본은 파일/DB/브라우저 어디에도 저장하지 않으며, 브라우저(`localStorage`)에는 최근
검색어 텍스트만 저장됩니다. API 연결에 실패하면 예시(샘플) 데이터가 화면에 표시될
수 있습니다.

## 실행 방법

이 프로젝트를 확인하는 방법은 두 가지입니다.

1. **배포된 사이트로 바로 보기**: 위 "배포된 사이트" 주소(https://trend-top10.onrender.com/)에
   접속하면 됩니다. 별도 설치나 실행 없이 바로 확인할 수 있습니다.
2. **내 컴퓨터에서 직접 실행하기**: 코드를 수정하거나 로컬에서 테스트하고 싶다면 아래
   "로컬 실행 방법"을 따르세요.

### 로컬 실행 방법 (초보자용)

`index.html`은 화면에 보여줄 데이터를 `server.js`(API 서버)에게 요청해서 받아옵니다.
그래서 아래 순서로 실행해야 정상적으로 데이터가 보입니다.

1. 이 폴더에서 터미널(PowerShell)을 열고 아래 명령어로 서버를 먼저 실행합니다.
   ```
   node server.js
   ```
   `가짜 API 서버 실행 중: http://localhost:3000/api/trends` 메시지가 보이면 성공입니다.
2. 서버를 켜둔 상태로 `index.html` 파일을 더블클릭해서 브라우저로 엽니다.

만약 서버를 켜지 않고 `index.html`만 열면, 데이터를 받아오지 못했다는 안내 배너가
화면에 표시됩니다.

### 배포된(Render) 사이트 주소 확인 방법 (초보자용)

지금 배포되어 있는 주소는 위에 적힌 https://trend-top10.onrender.com/ 이지만, 나중에
Render에서 서비스 이름을 바꾸거나 새로 배포해서 주소가 궁금해지면 아래처럼 확인하세요.

1. [render.com](https://render.com)에 로그인합니다.
2. 대시보드에서 이 프로젝트의 서비스(예: `trend-top10`)를 클릭합니다.
3. 서비스 이름 아래(또는 화면 상단)에 `https://서비스이름.onrender.com` 형태의 주소가
   표시됩니다. 이 주소가 실제로 배포된 사이트 주소입니다.
4. 이 주소를 브라우저에 입력하면 실제 서비스 화면이 보입니다.

- **상단 제목 영역**: "트렌드 TOP 10" 제목과 부제, 오늘 날짜(기준 날짜)를 자동으로 표시
- **데이터 안내 배너**: 네이버 API 기반 데이터 또는 예시(샘플) 데이터가 표시될 수 있음을
  헤더 바로 아래 눈에 띄는 배너로 안내

  - 가장 핫한 카테고리: 전체 데이터 중 가장 많이 등장하는 카테고리
  - 급상승 키워드: 검색 트렌드 데이터 중 순위가 가장 많이 오른 항목
  - 신규 진입 항목: 순위가 새로 등장(`new`)한 항목 개수
- **카테고리 선택 버튼**: 전체 / 패션의류 / 뷰티 / 생활용품 / 식품 / 디지털가전 / 스포츠레저
  버튼을 눌러 두 표를 동시에 필터링
- **쇼핑 검색 상품 TOP 10 / 검색 트렌드 TOP 10 표**: 순위 / 항목 / 카테고리 / 변동 4개 열로 구성
  - 변동 열은 상승(▲) / 하락(▼) / 변동없음 / 신규 진입(NEW)을 구분해서 표시
  - 1위·2위·3위 행은 배경색으로 강조 (1위가 가장 눈에 띄게, 2·3위는 약하게)
- **모바일 반응형**: 화면이 좁아지면 두 표가 가로가 아니라 세로로 쌓이도록 처리
- **API 서버 (`server.js`)**: `index.html`이 고정된 배열 대신 서버에 요청해서 데이터를
  받아오도록 구조를 바꿨습니다 (`GET /api/trends`). 서버가 꺼져 있으면 화면에 안내
  배너가 표시됩니다.
- **네이버 API 연동 준비 (용도별 키 분리)**: `server.js`가 **쇼핑인사이트(쇼핑 검색
  상품용)**와 **검색어트렌드(검색 트렌드용)** API 키를 각각 따로 읽어오는 구조를
  만들어뒀습니다.
  - `NAVER_SHOPPING_CLIENT_ID` / `NAVER_SHOPPING_CLIENT_SECRET`: 쇼핑 검색 상품(쇼핑인사이트)용
  - `NAVER_SEARCH_TREND_CLIENT_ID` / `NAVER_SEARCH_TREND_CLIENT_SECRET`: 검색 트렌드(검색어트렌드)용
  - 위 값이 없으면 기존 `NAVER_CLIENT_ID` / `NAVER_CLIENT_SECRET`을 공용으로 사용합니다(fallback).
  - `GET /api/naver-ready`로 두 API가 각각 준비됐는지 확인할 수 있습니다.
    ```json
    { "shoppingInsightReady": true, "searchTrendReady": false }
    ```

## 네이버 API 키 발급 및 환경변수 설정 방법 (초보자용)

아직 이 프로젝트는 실제로 네이버 API를 호출하지 않지만, 나중에 연동할 때를 대비해서
키를 미리 발급받고 환경변수로 설정하는 방법을 정리해뒀습니다.

네이버 API는 기능(상품)별로 따로 신청하기 때문에, **구매 순위(쇼핑인사이트)**와
**검색 순위(검색어트렌드)**를 각각 신청하면 Client ID/Secret이 서로 다른 키 "두 쌍"으로
나올 수도 있고, 신청 방식에 따라 하나의 애플리케이션에 두 API를 함께 등록해서 키 "한 쌍"만
받을 수도 있습니다. 두 경우 모두 지원하도록 만들어뒀으니, 아래에서 내 상황에 맞는 방법을
따라 하면 됩니다.

### 1. 네이버 개발자센터에서 API 키 발급받기

1. 네이버 개발자센터(developers.naver.com)에 접속해서 네이버 계정으로 로그인합니다.
2. "Application" 메뉴에서 애플리케이션을 새로 등록합니다.
3. 사용할 API를 선택합니다.
   - 구매 순위용: **쇼핑인사이트** API
   - 검색 순위용: **검색어트렌드** API
   - 둘 다 필요하면 신청 화면에서 두 API를 모두 체크해도 되고, 따로따로 애플리케이션을
     만들어도 됩니다.
4. 등록이 끝나면 **Client ID**와 **Client Secret** 값을 발급받습니다.
   - 이 값들은 비밀번호와 같습니다. 다른 사람에게 공유하거나 코드에 직접 적지 마세요.

### 2. `.env` 파일 채우기 — 키가 한 쌍인 경우

하나의 애플리케이션에서 두 API를 함께 등록해서 **Client ID/Secret을 한 쌍만** 받았다면,
`.env.example`에서 아래 두 줄의 주석(`#`)만 지우고 값을 채우면 됩니다. 나머지 4개 줄은
그대로 주석 처리된 상태로 두거나 지워도 괜찮습니다.

```
NAVER_CLIENT_ID=발급받은_클라이언트_ID
NAVER_CLIENT_SECRET=발급받은_클라이언트_시크릿
```

`server.js`는 용도별 환경변수(`NAVER_SHOPPING_...`, `NAVER_SEARCH_TREND_...`)가 없으면
이 공용 키를 구매/검색 두 곳 모두에 자동으로 사용합니다.

### 3. `.env` 파일 채우기 — 키가 두 쌍인 경우

구매 순위용과 검색 순위용 키를 **따로** 발급받았다면, 용도에 맞게 아래 4개 값을 각각
채워 넣습니다.

```
NAVER_SHOPPING_CLIENT_ID=쇼핑인사이트_클라이언트_ID
NAVER_SHOPPING_CLIENT_SECRET=쇼핑인사이트_클라이언트_시크릿

NAVER_SEARCH_TREND_CLIENT_ID=검색어트렌드_클라이언트_ID
NAVER_SEARCH_TREND_CLIENT_SECRET=검색어트렌드_클라이언트_시크릿
```

이렇게 용도별로 값을 채우면, 위의 공용 `NAVER_CLIENT_ID` / `NAVER_CLIENT_SECRET`이 있어도
용도별 값이 우선 사용됩니다.

### 4. 공통: `.env` 파일 준비 및 주의사항

1. 아직 `.env` 파일이 없다면, 이 프로젝트 폴더에 있는 `.env.example` 파일을 복사해서
   이름을 `.env`로 바꿉니다.
2. 위 2번 또는 3번 방법으로 값을 채워 넣고 저장합니다.
3. `.env` 파일은 절대 다른 사람과 공유하거나 깃허브 같은 곳에 올리지 마세요. 이 파일은
   내 컴퓨터에만 있어야 하는 비밀 파일입니다.

`server.js`는 실행될 때 `.env` 파일이 있으면 자동으로 읽어서 환경변수로 등록합니다
(별도의 외부 라이브러리 설치 없이 동작합니다).

### 5. 환경변수가 잘 설정됐는지 확인하기

1. 터미널에서 서버를 실행합니다.
   ```
   node server.js
   ```
2. 브라우저나 터미널에서 `http://localhost:3000/api/naver-ready` 주소에 접속합니다.
3. 아래처럼 두 값이 각각 표시됩니다.
   ```json
   { "shoppingInsightReady": true, "searchTrendReady": true }
   ```
   - `shoppingInsightReady`가 `true`면 구매 순위(쇼핑인사이트)용 키가 준비된 것입니다.
   - `searchTrendReady`가 `true`면 검색 순위(검색어트렌드)용 키가 준비된 것입니다.
   - `false`가 보이면 해당 용도의 `.env` 값(또는 공용 `NAVER_CLIENT_ID`/`SECRET`)을
     다시 확인해야 합니다.

> ⚠️ **다시 한번 강조**: 네이버 API 키(Client ID)나 Secret 값은 절대 `index.html`
> (또는 다른 클라이언트/브라우저 코드)에 직접 적지 마세요. `index.html`은 누구나
> "페이지 소스 보기"로 내용을 볼 수 있기 때문에, 여기에 키를 적으면 그 즉시 유출됩니다.
> API 키와 Secret은 반드시 서버 쪽 환경변수(`.env` 파일)로만 보관해야 합니다.

## v0.7 — 실제 네이버 API 호출 테스트

`index.html`은 이번 버전에서 **아직 수정하지 않았습니다.** 화면이 실제로 사용하는
`/api/trends`도 여전히 예전처럼 예시(가짜) 데이터를 그대로 돌려줍니다.

이번 버전에서는 `server.js`에 "테스트 전용" 주소 2개를 추가해서, 네이버 API 키가
잘 작동하는지 미리 확인만 해볼 수 있게 했습니다. 두 주소 모두 서버(`server.js`)가
네이버 서버에 직접 요청을 보내고, 받아온 응답을 거의 그대로 화면에 보여줍니다.

### 테스트 전 준비

1. `.env` 파일에 `NAVER_SEARCH_TREND_CLIENT_ID/SECRET`(검색어트렌드용) 또는
   `NAVER_SHOPPING_CLIENT_ID/SECRET`(쇼핑인사이트용) 값이 채워져 있어야 합니다.
   (둘 다 없으면 공용 `NAVER_CLIENT_ID`/`NAVER_CLIENT_SECRET`을 대신 사용합니다.)
   자세한 설정 방법은 위 "네이버 API 키 발급 및 환경변수 설정 방법" 절을 참고하세요.
2. 터미널에서 서버를 실행합니다.
   ```
   node server.js
   ```

## v0.8 — 네이버 API 응답을 화면 데이터에 연결

`index.html`이 실제로 데이터를 요청하는 `GET /api/trends`가 이제 네이버 API를 직접
호출합니다. `index.html` 자체는 이번에도 API 키/Secret을 전혀 담고 있지 않습니다
(키는 여전히 `server.js`와 `.env`에만 있습니다). 대신 아래처럼 화면에 보이는 문구를
사실과 다르지 않게 바꿨습니다.

- "구매 순위 TOP 10" → **"쇼핑 검색 상품 TOP 10"**
- "검색 순위 TOP 10" → **"검색 트렌드 TOP 10"**
- 예시 데이터 배너 → **"네이버 API 기반 데이터 또는 예시 데이터가 표시될 수 있습니다"**로 변경

### `/api/trends`가 하는 일

1. v0.7 테스트 때 썼던 것과 같은 키워드("선크림", "양산", "여름원피스")로 네이버
   검색어트렌드 API와 쇼핑인사이트 API를 각각 호출합니다.
2. 호출이 성공하면, 응답에 담긴 관심도 수치(`ratio`)를 이용해 키워드를 순위(rank)로
   정렬하고, 화면 표가 쓰는 `{ rank, name, category, change }` 모양으로 바꿔서
   돌려줍니다.
   - `searchData`: 검색어트렌드 API 결과로 만든 "검색 트렌드 TOP 10" 표 데이터
   - `shoppingProductData`: 쇼핑인사이트 API 결과로 만든 "쇼핑 검색 상품 TOP 10" 표 데이터
3. 호출이 실패하거나(키가 없거나 틀림, 네이버 서버 오류 등) 응답에 쓸만한 데이터가
   없으면, 그 목록만 기존 예시(가짜) 데이터로 대신 채웁니다. 검색/쇼핑 중 한쪽만
   실패해도 나머지 한쪽은 실제 데이터를 그대로 보여줍니다.
4. 어떤 경우에도 네이버 API 응답이나 예시 데이터를 파일이나 DB에 저장하지 않고,
   요청이 들어올 때마다 새로 계산해서 그때그때 돌려줍니다.

### ⚠️ 주의사항: API 응답을 왜곡하지 않기

- 표에 보이는 **순위(rank)**는 네이버가 공식적으로 발표하는 "실시간 쇼핑/검색 순위"가
  아니라, 우리가 미리 정해둔 테스트 키워드 몇 개만 놓고 관심도(`ratio`) 평균이 높은
  순서로 매긴 것입니다. 키워드 종류를 늘리거나 바꾸면 순위도 달라집니다.
- **변동(change)** 값도 네이버가 제공하는 값이 아니라, 최근 7일 평균과 이전 7일 평균의
  관심도 차이를 서버가 직접 계산한 근사치입니다(`server.js`의 `computeChangeFromSeries`
  함수). "선크림이 실제로 검색 3위다" 같은 식으로 공식 통계처럼 소개하지 않도록
  주의해야 합니다.
- 카테고리 값(뷰티/패션의류/생활용품 등)도 네이버 응답에 들어있는 값이 아니라, 테스트
  키워드 각각에 대해 코드에서 미리 정해둔 매핑(`TREND_TEST_KEYWORD_CATEGORY`)입니다.
  키워드를 추가할 때는 이 매핑도 함께 채워줘야 합니다.
- 네이버 원본 응답의 숫자(`ratio` 등) 자체는 가공하지 않고 그대로 계산에만 사용하며,
  임의로 수치를 부풀리거나 순서를 조작하지 않습니다.

1. 이 폴더에서 터미널(PowerShell)을 열고 서버를 실행합니다.
   ```
   node server.js
   ```
2. 서버를 켜둔 상태로 `index.html` 파일을 더블클릭해서 브라우저로 엽니다.
3. 화면 상단 배너에 "네이버 API 기반 데이터 또는 예시 데이터가 표시될 수 있습니다"
   문구가 보이면 정상입니다.
4. "쇼핑 검색 상품 TOP 10"과 "검색 트렌드 TOP 10" 표에 "선크림", "양산", "여름원피스"
   같은 이름이 보이면 실제 네이버 API 데이터가 연결된 것입니다. (`.env`에 키가 없거나

   채워두지 않은 경우에도 이 메시지가 나오며(다만 `status`가 500으로 표시됩니다),

## v0.9 — 데이터 출처와 API 상태 표시 개선

v0.8까지는 화면에 실제 네이버 데이터가 나오는지 예시 데이터가 나오는지 안내 배너의
일반적인 문구로만 짐작할 수 있었습니다. v0.9에서는 **어떤 표가 실제 데이터고 어떤
표가 예시 데이터인지**를 표마다 바로 확인할 수 있도록 안내를 구체화했습니다.
`index.html`에는 이번에도 API 키/Secret을 전혀 추가하지 않았습니다.

### 안내 문구 개선

헤더 아래 안내 배너 문구를 아래 세 가지 내용을 모두 포함하도록 다시 정리했습니다.

> ℹ️ 이 화면은 네이버 API 응답을 바탕으로 만든 참고용 트렌드 화면입니다. 표시되는
> 순위와 변동값은 네이버 공식 구매 순위가 아니라, API의 관심도(ratio) 데이터를
> 화면용으로 계산한 값입니다. API 연결에 실패하면 예시 데이터가 표시될 수 있습니다.

### 표마다 데이터 출처 표시

"쇼핑 검색 상품 TOP 10"과 "검색 트렌드 TOP 10" 제목 옆에 작은 배지를 추가해서, 그
표가 실제 네이버 API 데이터인지 예시 데이터인지 바로 보이게 했습니다.

- 초록색 배지 **"네이버 API 데이터"**: 해당 목록이 네이버 API 호출에 성공해서 실제
  데이터로 채워졌다는 뜻입니다.
- 빨간색 배지 **"예시 데이터 표시 중"**: 해당 목록의 API 호출이 실패했거나 아직 서버
  응답을 받기 전이라, 예시(가짜) 데이터가 대신 표시되고 있다는 뜻입니다.

두 표는 서로 독립적으로 표시되므로, 예를 들어 검색어트렌드 API만 실패했다면 "검색
트렌드 TOP 10"에만 빨간 배지가 뜨고 "쇼핑 검색 상품 TOP 10"은 초록 배지를 유지합니다.

### `/api/trends` 응답에 출처 필드 추가

```json
{
  "searchData": [...],
  "searchDataSource": "naver",
  "shoppingProductData": [...],
  "shoppingProductSource": "naver",
  "categories": [...],
  "baseDate": "..."
}
```

- `searchDataSource` / `shoppingProductSource` 값은 각각 `"naver"`(실제 API 데이터) 또는
  `"mock"`(예시 데이터) 중 하나입니다.
- 기존 기능(카테고리 필터, 오늘의 트렌드 요약, 1~3위 강조)은 그대로 유지됩니다.

### 브라우저에서 확인하는 방법 (초보자용)

1. 터미널에서 서버를 실행합니다.
   ```
   node server.js
   ```
2. 서버를 켜둔 상태로 `index.html`을 더블클릭해서 브라우저로 엽니다.
3. 헤더 아래 안내 문구가 "이 화면은 네이버 API 응답을 바탕으로 만든 참고용 트렌드

4. "쇼핑 검색 상품 TOP 10"과 "검색 트렌드 TOP 10" 제목 옆 배지를 확인합니다.
   - 둘 다 초록색 "네이버 API 데이터"면 두 API 모두 정상 연결된 것입니다.
   - 하나라도 빨간색 "예시 데이터 표시 중"이면, 그 API 호출만 실패했다는 뜻입니다.

   배지만 빨간색 "예시 데이터 표시 중"으로 바뀌고, "쇼핑 검색 상품 TOP 10"은 그대로

## v1.0 — 사용자 키워드 입력 기능

이제 화면에서 직접 궁금한 키워드를 입력해서 그 키워드로 네이버 트렌드를 조회해볼 수
있습니다. `index.html`에는 이번에도 API 키/Secret을 전혀 추가하지 않았습니다.

### 화면에 추가된 것

- "카테고리 선택" 아래에 **"관심 키워드로 트렌드 조회"** 영역이 새로 생겼습니다.
  - 입력 칸 3개(기본값: "선크림", "양산", "여름원피스")와 **[트렌드 조회]** 버튼이 있습니다.
  - 입력 칸을 원하는 키워드로 바꾸고 버튼을 누르면, 그 키워드들로 두 표(쇼핑 검색 상품 /
    검색 트렌드)가 다시 조회됩니다.
  - 입력 칸을 모두 비워두고 버튼을 누르면 기본 키워드(선크림, 양산, 여름원피스)로 조회합니다.
- 버튼을 누르면 데이터를 받아오는 동안 **"⏳ 데이터를 불러오는 중입니다..."** 문구가
  보이고, 응답이 올 때까지 버튼이 잠시 비활성화됩니다.
- 키워드 입력이 잘못된 경우(칸 일부만 비어있거나, 20자를 넘거나, 4개 이상 입력한 경우
  등) 안내 문구가 빨간 글씨로 표시되고, 화면의 기존 표는 그대로 유지됩니다.
- 기존 기능(카테고리 필터, 오늘의 트렌드 요약, 데이터 출처 배지, 1~3위 강조)은 모두
  그대로 유지됩니다.

### `/api/trends?keywords=...` 동작 방식

- `keywords` 쿼리값은 쉼표(,)로 구분한 키워드 목록입니다. 예: `?keywords=선크림,양산,여름원피스`
- `keywords` 값이 없거나 비어 있으면 기본 키워드(선크림, 양산, 여름원피스)를 사용합니다.
- 키워드는 **최대 3개**까지만 허용됩니다. 그보다 많으면 아래처럼 에러를 돌려줍니다.
  ```json
  {
    "status": 400,
    "message": "키워드는 최대 3개까지만 입력할 수 있습니다.",
    "detail": "입력한 키워드 개수: 4개 (a, b, c, d)"
  }
  ```
- 키워드 하나의 길이는 **20자**를 넘을 수 없습니다. 넘으면 `status: 400`과 함께 어떤
  키워드가 너무 긴지 안내합니다.
- 쉼표 사이에 빈 값이 있으면(예: `선크림,,양산`) 형식 오류로 안내합니다.
- 유효성 검사를 통과한 키워드로 네이버 검색어트렌드/쇼핑인사이트 API를 호출하고, 이후
  동작(성공 시 실제 데이터로 변환, 실패 시 목록별 예시 데이터 fallback, 결과 저장하지
  않음)은 v0.8/v0.9와 동일합니다. 사용자가 입력한 키워드가 미리 정해둔 카테고리 매핑에
  없으면 카테고리는 "생활용품"으로 표시됩니다.

### 브라우저에서 확인할 항목 (초보자용)

1. 터미널에서 서버를 실행합니다.
   ```
   node server.js
   ```
2. 서버를 켜둔 상태로 `index.html`을 더블클릭해서 브라우저로 엽니다. 페이지가 열리자마자
   기본 키워드로 자동 조회가 한 번 실행됩니다.
3. "관심 키워드로 트렌드 조회" 입력 칸 3개에 원하는 키워드(예: "노트북", "마우스",
   "키보드")를 입력하고 **[트렌드 조회]** 버튼을 눌러봅니다.
4. 버튼을 누르는 순간 "⏳ 데이터를 불러오는 중입니다..." 문구가 잠깐 보였다가 사라지고,

   - 아무 칸에나 20자가 넘는 아주 긴 텍스트 입력 후 조회 (너무 긴 키워드 안내)
6. 입력 칸 3개를 모두 지우고 조회하면, 에러 없이 기본 키워드(선크림/양산/여름원피스)

7. 서버(`node server.js`를 실행한 터미널 창)를 잠깐 꺼둔 채로 버튼을 눌러서, "⏳ 데이터를
   불러오는 중입니다..." 문구가 사라진 뒤 기존처럼 "⚠️ 서버 연결 실패" 배너가 뜨는지도

## v1.1 — 조회 기준 표시 영역 추가

이제 [트렌드 조회] 버튼을 누르면(또는 페이지를 처음 열어서 자동 조회가 실행되면), 이번
조회에 실제로 어떤 기준이 쓰였는지를 화면에서 바로 확인할 수 있습니다. `index.html`에는
이번에도 API 키/Secret을 전혀 추가하지 않았습니다.

### 화면에 추가된 것

"관심 키워드로 트렌드 조회" 영역 바로 아래에 **"조회 기준"** 섹션이 새로 생겼습니다.
트렌드 조회 결과를 받기 전에는 보이지 않다가, 첫 조회 응답을 받으면 아래 4가지 정보가
채워지면서 나타납니다.

1. **조회 키워드**: 이번 조회에 실제로 사용된 키워드 목록 (예: "선크림, 양산, 여름원피스")
2. **조회 기간**: 네이버 API에 실제로 요청한 조회 기간 (예: "2026-06-06 ~ 2026-07-05")
3. **데이터 출처**: 네이버 데이터랩 API(검색어트렌드 · 쇼핑인사이트)에서 가져온다는 안내와,
   표마다 실제 데이터/예시 데이터 여부는 표 제목 옆 배지에서 확인할 수 있다는 설명
4. **순위 산정 방식**: 화면의 순위가 네이버 API의 관심도(ratio) 평균값을 기준으로 계산한
   "화면용 참고 순위"이며, 네이버 공식 구매 순위가 아니라는 설명

기존 기능(키워드 입력, 카테고리 필터, 오늘의 트렌드 요약, 데이터 출처 배지, 1~3위 강조)은
모두 그대로 유지됩니다.

### `/api/trends` 응답에 추가된 값

```json
{
  "searchData": [...],
  "searchDataSource": "naver",
  "shoppingProductData": [...],
  "shoppingProductSource": "naver",
  "categories": [...],
  "baseDate": "...",
  "activeKeywords": ["선크림", "양산", "여름원피스"],
  "dateRange": { "startDate": "2026-06-06", "endDate": "2026-07-05" }
}
```

- `activeKeywords`: 이번 요청에서 실제로 사용된 키워드 배열입니다. 사용자가 입력 칸에
  키워드를 채워서 조회하면 그 키워드가, 입력 칸을 비워두면 기본 키워드(선크림, 양산,
  여름원피스)가 담깁니다.
- `dateRange`: 네이버 API 호출에 실제로 사용한 조회 기간(`startDate` ~ `endDate`,
  `YYYY-MM-DD` 형식)입니다. 최근 30일(어제까지) 기준으로 서버가 자동 계산합니다.
- 두 값 모두 서버가 계산해서 그대로 알려주는 값이며, 화면(index.html)은 이 값을 받아서
  보여주기만 할 뿐 직접 계산하지 않습니다.

### 브라우저에서 확인할 항목 (초보자용)

1. 터미널에서 서버를 실행합니다.
   ```
   node server.js
   ```
2. 서버를 켜둔 상태로 `index.html`을 더블클릭해서 브라우저로 엽니다.
3. 페이지가 열리고 자동 조회가 끝나면, "관심 키워드로 트렌드 조회" 영역 아래에 "조회 기준"
   섹션이 나타나는지 확인합니다.
4. "조회 기준" 섹션에 아래 4가지가 모두 채워져 있는지 확인합니다.
   - 조회 키워드: "선크림, 양산, 여름원피스"
   - 조회 기간: 오늘 날짜 기준 최근 30일 범위 (예: "2026-06-06 ~ 2026-07-05")
   - 데이터 출처: 네이버 데이터랩 API 관련 설명 문구
   - 순위 산정 방식: 참고 순위이며 공식 구매 순위가 아니라는 설명 문구
5. 입력 칸에 다른 키워드(예: "노트북", "마우스", "키보드")를 넣고 [트렌드 조회]를 눌러본
   뒤, "조회 기준"의 **조회 키워드** 값이 방금 입력한 키워드로 바뀌는지 확인합니다.
6. 브라우저 주소창에 `http://localhost:3000/api/trends`를 직접 입력해서 응답 JSON에
   `activeKeywords`와 `dateRange` 값이 포함되어 있는지 확인해봐도 됩니다.

### v1.1 백업 생성

조회 기준 표시 영역까지 정상 동작하는 v1.1 시점을 스냅샷으로 남겨두기 위해

## v1.2 — 사용성 개선

이번 버전은 새 기능을 추가하기보다, **기존 화면을 더 쉽고 편하게 쓸 수 있도록 다듬는 것**이
목표입니다. `server.js`와 네이버 API 연동 구조는 그대로 두었고, `index.html`에도 API
키/Secret은 이번에도 전혀 추가하지 않았습니다.

### 키워드 입력 영역 안내 개선

- 입력 칸 위에 "궁금한 상품이나 검색어를 입력하고 [트렌드 조회] 버튼을 누르면..." 같은
  쉬운 설명 문구를 추가해서, 처음 보는 사람도 이 영역이 무엇을 하는 곳인지 바로 알 수
  있게 했습니다.
- "최대 3개 · 칸마다 20자 이하로 입력해 주세요"라는 조건 안내를 입력 칸 바로 아래에
  작은 회색 글씨로 별도 표시했습니다. (기존의 "기본 키워드로 조회합니다" 안내 문구와는
  분리해서, 조건과 기본값 안내를 각각 더 짧고 읽기 쉽게 나눴습니다.)

### 서버 연결 실패 배너 개선

서버가 꺼져 있어서 데이터를 못 받아왔을 때 뜨는 빨간 배너에 "PowerShell에서
`node server.js`를 실행했는지 확인해 주세요"라는 문구를 추가했습니다. 서버를 처음
실행해보는 초보자도 무엇을 확인해야 할지 바로 알 수 있습니다.

### 모바일 화면에서 키워드 입력 영역 개선

이전에는 화면이 좁아져도 키워드 입력 칸 3개와 [트렌드 조회] 버튼이 계속 가로로 나란히
배치되어 있어서, 모바일에서는 칸이 너무 작아지고 버튼을 누르기 불편했습니다. 이제 화면
폭이 700px 이하로 좁아지면 입력 칸 3개와 버튼이 각각 세로로 쌓이면서 폭 전체를 차지하도록
바꿨습니다. (표 영역이 세로로 쌓이는 기존 모바일 대응과 같은 방식입니다.)

### "조회 기준" 영역을 카드 형태로 정리

v1.1에서 추가한 "조회 기준" 영역이 위에서 아래로 긴 목록(리스트) 형태라 화면을 많이
차지했습니다. v1.2에서는 이 4가지 정보(조회 키워드 / 조회 기간 / 데이터 출처 / 순위
산정 방식)를 2x2 격자(그리드)로 배치된 작은 카드들로 바꿔서 한눈에 훑어보기 쉽게
정리했습니다. 화면이 좁아지면(모바일) 카드가 1열로 쌓입니다. 데이터 출처와 순위 산정
방식 설명 문구도 더 짧고 간결하게 다듬었습니다.

### 기존 기능은 모두 그대로 유지

키워드 조회, 카테고리 필터, 오늘의 트렌드 요약, 데이터 출처 배지, 1~3위 강조, API 호출
실패 시 예시 데이터로 대체(fallback)하는 기능은 이번 버전에서 전혀 바뀌지 않았습니다.

### 브라우저에서 확인할 항목 (초보자용)

1. 터미널에서 서버를 실행합니다.
   ```
   node server.js
   ```
2. 서버를 켜둔 상태로 `index.html`을 더블클릭해서 브라우저로 엽니다.
3. "관심 키워드로 트렌드 조회" 영역에 쉬운 설명 문구와 "최대 3개 · 칸마다 20자 이하로
   입력해 주세요"라는 작은 안내가 보이는지 확인합니다.
4. "조회 기준" 영역이 긴 목록이 아니라 2x2 형태의 작은 카드들로 보이는지 확인합니다.
5. 브라우저 창 크기를 좁혀보거나(또는 개발자 도구의 모바일 화면 보기로 전환해서) 아래를

   - 키워드 입력 칸 3개와 [트렌드 조회] 버튼이 가로 한 줄이 아니라 세로로 쌓이며, 각각

   강조 배경색)이 이전과 똑같이 잘 동작하는지도 함께 확인합니다.

### v1.2 백업 생성

## v1.3 — 결과 요약 복사 기능

이제 화면에 표시된 결과를 버튼 하나로 텍스트 복사해서, 메신저나 메모장 등에 바로
붙여넣을 수 있습니다. `server.js`는 이번 버전에서 전혀 수정하지 않았고, `index.html`에도
API 키/Secret은 이번에도 추가하지 않았습니다.

### 화면에 추가된 것

"결과 요약 복사" 섹션이 새로 생겼습니다. 위치는 "조회 기준" 섹션과 순위 표 사이입니다.

- **[결과 요약 복사]** 버튼: 누르면 현재 화면에 보이는 내용을 정리한 텍스트를 클립보드에
  복사합니다.
- 복사에 성공하면 버튼 아래에 초록색 글씨로 **"✅ 결과 요약이 복사되었습니다"**가
  표시됩니다.
- 복사에 실패하면(브라우저가 클립보드 접근을 막은 경우 등) 빨간색 글씨로 무엇을 확인하면
  되는지 안내하는 문구가 표시됩니다.

### 복사되는 내용

버튼을 누른 시점에 화면에 실제로 보이는 내용을 그대로 텍스트로 옮깁니다. **카테고리
필터가 적용되어 있으면(예: "뷰티"만 선택한 상태) 그 카테고리로 걸러진 목록만
복사됩니다** — 화면에 안 보이는 다른 카테고리 항목은 복사되지 않습니다.

1. 조회 키워드
2. 조회 기간
3. 데이터 출처 (쇼핑 검색 상품 / 검색 트렌드 각각 네이버 API 데이터인지 예시 데이터인지)
4. 쇼핑 검색 상품 TOP 10 (현재 카테고리 필터 기준)
5. 검색 트렌드 TOP 10 (현재 카테고리 필터 기준)
6. 순위 산정 방식 안내 문구
7. "공식 구매 순위가 아닌 참고용 계산 결과입니다." 문구

복사되는 텍스트는 대략 아래와 같은 모양입니다.

```
[트렌드 TOP 10 결과 요약]

## v1.4 — 최근 조회 키워드 기억 기능

이제 마지막으로 조회했던 키워드를 브라우저가 기억해뒀다가, 페이지를 다시 열었을 때

`index.html`에도 API 키/Secret은 이번에도 추가하지 않았습니다.

### `localStorage`가 무엇인가요? (초보자용)

`localStorage`는 **브라우저가 내 컴퓨터(정확히는 지금 쓰고 있는 그 브라우저)에만 저장해두는
작은 저장 공간**입니다. 서버나 다른 사람의 컴퓨터로 전송되지 않고, 오직 "이 브라우저에서
이 사이트를 다시 열었을 때" 꺼내볼 수 있는 값입니다. 브라우저 저장소를 직접 지우거나
시크릿/프라이빗 모드로 열면 값이 남아있지 않을 수 있습니다.

이 프로젝트에서는 `localStorage`에 **사용자가 입력한 키워드 텍스트**만 저장합니다.
네이버 API가 돌려준 순위·변동 같은 실제 데이터는 이 기능과 전혀 관계없이 이전 버전부터
지금까지 한 번도 저장한 적이 없고, 이번에도 저장하지 않습니다. (서버가 파일이나 DB에
아무것도 저장하지 않는다는 점도 이전 버전과 동일합니다.)

### 화면에 추가된 것

- **[트렌드 조회]** 버튼을 누르면, 그 순간 입력 칸에 있던 키워드가 `localStorage`에
  저장됩니다. 다음에 이 페이지를 다시 열면 그 키워드가 입력 칸에 자동으로 채워집니다.
- **[기본 키워드로 되돌리기]** 버튼이 새로 생겼습니다. 누르면:
  - 입력 칸 3개가 기본 키워드(선크림, 양산, 여름원피스)로 되돌아갑니다.
  - `localStorage`에 저장해뒀던 최근 키워드도 함께 삭제됩니다.
- 키워드 입력 칸 아래에 작은 안내 문구가 추가됐습니다.
  > 🔒 최근 조회 키워드는 이 브라우저에만 저장되며, 네이버 API 응답 데이터는 저장하지
  > 않습니다.

데이터 출처는 쇼핑 검색 상품 - 네이버 API 데이터, 검색 트렌드 - 네이버 API 데이터입니다.
이 결과는 네이버 공식 구매 순위가 아닌 참고용 계산 결과입니다.
```

요약 문장에는 항상 아래 내용이 포함됩니다.

2. 조회 기간

5. 가장 많이 등장한 카테고리
6. 데이터 출처
7. "네이버 공식 구매 순위가 아닌 참고용 계산 결과입니다" 안내

### 카테고리 필터와 함께 갱신됨

이 문장은 **항상 "지금 화면에 보이는" 데이터를 기준으로** 계산됩니다. 카테고리 버튼을
눌러 예를 들어 "뷰티"만 선택하면, 1위 항목과 "가장 많이 등장한 카테고리"도 뷰티
카테고리 안에서 다시 계산되어 문장이 즉시 바뀝니다. 만약 선택한 카테고리에 해당하는
데이터가 하나도 없으면 "해당 카테고리 데이터 없음"으로 표시됩니다.

### 예시 데이터가 섞여 있을 때

자동으로 덧붙습니다.

## v1.7 — 조회 중/오류 UX 개선

이번 버전은 새 데이터나 새 기능보다, **조회 중일 때와 뭔가 잘못됐을 때 화면이 무슨
상태인지 더 분명하게 보여주는 것**이 목표입니다. `server.js`는 이번 버전에서 전혀
수정하지 않았고, `index.html`에도 API 키/Secret은 이번에도 추가하지 않았습니다.

### 조회 중 표시 개선

- [트렌드 조회] 버튼을 누르면 버튼 문구가 **"조회 중..."**으로 바뀌고, 응답이 올 때까지
  다시 눌리지 않도록 비활성화됩니다. (버튼이 원래 상태로 돌아오는 시점은 이전과 동일하게
  성공/실패와 상관없이 응답을 받은 직후입니다.)
- 같은 시간 동안 **순위 표 영역("쇼핑 검색 상품 TOP 10" / "검색 트렌드 TOP 10")**과
  **"결과 요약 문장" 영역**이 살짝 흐리게(블러) 보여서, "지금 화면 내용이 바뀌는 중"임을
  시각적으로 알 수 있습니다.
- 조회가 끝나면(성공이든 실패든) 흐림 효과가 사라지고 원래대로 선명하게 돌아옵니다.

### 안내 문구를 세 가지 상태로 명확히 구분

이전에는 "서버 연결 실패"와 "일부 데이터가 예시 데이터로 대체된 상태"가 잘 구분되지
않았습니다. v1.7부터는 아래 3가지 상태를 서로 다른 색과 문구로 구분합니다.

1. **입력 오류** (키워드 형식이 잘못된 경우): 입력 칸 아래에 빨간 글씨로
   **"⚠️ 입력 오류: ..."**가 표시됩니다. (기존과 같은 위치, 문구 맨 앞에 "입력 오류"를
   명시해서 다른 두 상태와 헷갈리지 않게 했습니다.)
2. **서버 연결 실패** (서버가 꺼져 있거나 네트워크 문제인 경우): 페이지 상단에 빨간
   배너로 **"⚠️ 서버 연결 실패: 화면에는 이전에 불러온 결과(또는 예시 데이터)가 그대로
   표시되고 있습니다..."**가 표시됩니다.
3. **API 일부 실패(fallback) 상태** (서버 응답 자체는 성공했지만, 네이버 API 호출
   일부가 실패해서 예시 데이터로 대체된 경우): 페이지 상단에 새로 생긴 노란 배너로
   **"ℹ️ 조회는 정상적으로 처리됐지만, 일부 데이터는 네이버 API 호출에 실패해 예시
   데이터로 대체되어 표시 중입니다..."**가 표시됩니다. (표 제목 옆 배지와 함께 보면
   어떤 표가 예시 데이터인지 바로 알 수 있습니다.)

### 실패해도 직전 성공 결과가 화면에 남습니다

입력 오류나 서버 연결 실패가 발생해도, 순위 표·데이터 출처 배지·조회 기준 카드·결과
요약 문장 등은 **직전에 성공했던 조회 결과 그대로** 남아 있습니다. (이 동작은 사실
v1.3~v1.6에서부터 실패 시 데이터를 따로 지우지 않는 방식으로 이미 만들어져 있었고,
v1.7에서 안내 문구를 더 명확히 다듬으면서 그대로 유지했습니다.)

- 페이지를 열자마자 **첫 조회부터 실패**해서 아직 성공한 결과가 하나도 없는 경우에는,
  미리 준비해둔 예시 데이터가 표시되고 표 제목 옆 배지에도 "예시 데이터 표시 중"이라고
  명확히 표시됩니다.
- **마지막 조회 시각**은 이전과 동일하게 조회가 성공했을 때만 갱신되고, 실패 시에는
  그대로 유지됩니다.
- **[결과 요약 복사]**도 이전과 동일하게, 지금 화면에 남아 있는 결과(성공한 마지막
  조회 결과, 또는 예시 데이터)를 기준으로 텍스트를 만듭니다.

### 기존 기능은 모두 그대로 유지

키워드 조회, 기본 키워드로 되돌리기, 최근 조회 키워드 기억, 카테고리 필터, 오늘의

## v1.8 — 간단 막대 차트 기능

이제 표 아래에 순위를 한눈에 비교할 수 있는 막대 차트가 함께 표시됩니다. 외부 차트
라이브러리나 CDN은 전혀 쓰지 않고, 순수 HTML/CSS/JavaScript만으로 만들었습니다.
`server.js`는 이번 버전에서 전혀 수정하지 않았고, `index.html`에도 API 키/Secret은
이번에도 추가하지 않았습니다.

### 화면에 추가된 것

"쇼핑 검색 상품 TOP 10"과 "검색 트렌드 TOP 10" 표 바로 아래에 각각 가로 막대 차트가
새로 생겼습니다. 막대 하나마다 아래 내용이 함께 표시됩니다.

- 순위 (예: "3위")
- 항목 이름
- 변동 표시 (▲ 상승 / ▼ 하락 / 변동없음 / NEW, 표에 쓰는 것과 같은 배지)
- 순위를 기준으로 계산한 상대 길이의 막대

### 막대 길이는 어떻게 계산하나요? (초보자용, 중요)

네이버 API가 실제로 돌려주는 관심도(ratio) 원본 수치는 서버(`server.js`)가 화면
(`index.html`)으로 내려주지 않습니다(순위로 정렬한 뒤 `rank`/`name`/`category`/`change`
값만 내려줍니다). 그래서 이 차트의 막대 길이는 **실제 ratio 수치가 아니라, 화면에 있는
순위(1~10)를 거꾸로 뒤집어 만든 상대 점수**입니다(1위가 가장 길고, 순위가 낮을수록
막대도 짧아집니다). 각 차트 위에 있는 작은 안내 문구
("막대 길이는 순위를 기준으로 계산한 상대 점수이며, 실제 관심도(ratio) 수치가
아닙니다.")가 바로 이 내용을 설명합니다. 순위 자체도 이미 "네이버 공식 구매 순위가
아닌 참고용 계산 결과"라는 점은 이전 버전과 동일합니다.

### 카테고리 필터, fallback과 함께 동작

- 카테고리 버튼을 누르면 표와 차트가 **동시에** 현재 화면 기준으로 다시 그려집니다.
  특정 카테고리를 선택하면 그 카테고리에 해당하는 항목만 차트에 남습니다.
- 네이버 API 호출 일부가 실패해서 예시 데이터가 표시 중이면(v1.7의 노란 "일부 데이터는

### 배포 전 주의사항 (초보자용)

1. **`.env` 파일은 절대 GitHub에 올리지 마세요.** `.gitignore`를 만들어뒀지만, 이미
   예전에 `.env`를 커밋한 적이 있다면 `.gitignore`만으로는 이미 올라간 기록까지

   확인하세요.)
2. **실제 키를 코드에 직접 적지 마세요.** `NAVER_SHOPPING_CLIENT_SECRET` 같은 값은
   항상 `.env` 파일에만 두고, `index.html`이나 `server.js` 코드 안에 직접 붙여넣지
   마세요.
3. **`.env.example`은 그대로 공유해도 안전합니다.** 실제 값이 아니라 "여기에 입력"
   같은 안내 문구만 있기 때문에, 다른 사람과 프로젝트를 공유할 때 `.env.example`은
   함께 올려도 되고, `.env`는 절대 올리면 안 됩니다.
4. **배포 후에도 `/api/naver-ready`로 키 설정 상태를 확인**할 수 있습니다. 이 주소는
   키가 "설정되어 있는지 여부(true/false)"만 알려주고 실제 키 값은 절대 보여주지
   않습니다.
5. **네이버 API 사용량 제한**을 확인하세요. 이 프로젝트는 방문할 때마다(또는 [트렌드
   조회]를 누를 때마다) 네이버 API를 새로 호출합니다. 방문자가 많아지면 네이버
   개발자센터에서 정해둔 일일 호출 한도를 넘길 수 있으니, 실제 배포 전에 한도를
   확인해두는 것이 좋습니다.
6. **화면 문구는 계속 "참고용"임을 명시해야 합니다.** 지금 배너와 요약 문장에 있는
   "네이버 공식 구매 순위가 아닌 참고용 계산 결과입니다" 안내를 앞으로도 지우지
   마세요. 이 문구가 있어야 이 화면이 네이버의 공식 통계가 아니라는 점을 사용자가
   오해하지 않습니다.

## Render 배포 전 최종 점검

`server.js`(가짜 API 서버)를 [Render](https://render.com)에 배포하기 전에 마지막으로
점검한 내용입니다. 실제 화면/기능은 바뀌지 않았고, 배포 환경에 맞춰 포트 처리 방식만
고쳤습니다.

### 점검 항목과 결과

| 점검 항목 | 결과 |
| --- | --- |
| `server.js`가 포트를 `process.env.PORT \|\| 3000` 방식으로 읽는가? | ⚠️ → ✅ 수정함 (기존에는 `server.listen(3000)`으로 고정되어 있었습니다) |
| `index.html`에 API 키/Secret이 있는가? | ✅ 없음 (다시 전체 검사, 이전과 동일하게 안전) |
| `.env`가 `.gitignore`에 의해 Git에서 제외되는가? | ✅ 포함되어 있음 (`.env` 항목이 그대로 있고, `.claude/` 항목도 추가되어 있음을 확인) |

### 실제로 바뀐 부분

`server.js` 맨 아래의 서버 실행 부분을 아래처럼 고쳤습니다.

```js
// 이전 (포트가 3000으로 고정되어 있어 Render에서는 동작하지 않음)
server.listen(3000, function () { ... });

// 이후 (Render가 알려주는 PORT 환경변수가 있으면 그 값을, 없으면 3000을 사용)
var PORT = process.env.PORT || 3000;
server.listen(PORT, function () { ... });
```

Render 같은 호스팅 서비스는 서버가 실제로 어떤 포트 번호를 써야 하는지 자기들이
결정한 뒤 `PORT`라는 환경변수로 알려줍니다. 포트 번호가 3000으로 고정되어 있으면
Render 위에서 서버가 아예 뜨지 않을 수 있어서, 이번에 `process.env.PORT || 3000`
방식으로 바꿨습니다. 내 컴퓨터에서 `node server.js`로 그냥 실행할 때는 `PORT`
환경변수가 없으므로 지금까지와 똑같이 3000번 포트로 실행됩니다(직접 실행해서
`http://localhost:3000`으로 확인했고, `PORT` 값을 넣었을 때도 그 포트로 정상 실행되는
것까지 확인했습니다).

### Render 배포 설정값 (초보자용)

[render.com](https://render.com)에서 "New +" → "Web Service"로 새 서비스를 만들 때
아래 값을 그대로 입력하면 됩니다.

| 설정 항목 | 입력할 값 |
| --- | --- |
| Service Type | **Web Service** |
| Language(Runtime) | **Node** |
| Branch | **main** |
| Build Command | `npm install` (또는 비워둬도 됩니다 — 이 프로젝트는 외부 패키지를 쓰지 않습니다) |
| Start Command | `node server.js` |

**환경변수(Environment Variables)** 설정 화면에 아래 4개 이름을 만들고, 각각 실제
네이버 API에서 발급받은 값을 입력합니다.

```

NAVER_SHOPPING_CLIENT_SECRET
NAVER_SEARCH_TREND_CLIENT_ID
NAVER_SEARCH_TREND_CLIENT_SECRET
```

(용도별 키 대신 공용 키 한 쌍만 있다면 `NAVER_CLIENT_ID`, `NAVER_CLIENT_SECRET` 두
개만 입력해도 됩니다. `.env.example`에 설명된 규칙과 동일합니다.)

> ⚠️ **중요**: 실제 API 키/Secret 값은 **GitHub 저장소(코드)나 `.env` 파일에 커밋해서
> 올리는 것이 아니라, 반드시 Render의 "Environment Variables" 설정 화면에 직접
> 입력**해야 합니다. GitHub에는 `.env.example`처럼 값이 비어있는 안내 파일만 올라가야
> 하고, `.gitignore`가 `.env` 파일 자체를 Git에서 제외해줍니다.

## v2.1 — Render 배포 화면 표시 문제 수정

Render에 배포한 주소(`https://trend-top10.onrender.com`)로 접속했을 때 "Not Found"가
뜨는 문제를 고쳤습니다. 원인은 `server.js`가 `/api/...` 같은 API 주소만 처리할 뿐,
메인 화면 주소(`/`)로 접속했을 때 `index.html`을 보여주는 코드가 없었기 때문입니다.
API 키/Secret은 이번에도 `index.html`에 넣지 않았고, `.env` 파일도 건드리지
않았습니다.

### 실제로 바뀐 부분

**1) `server.js`: `GET /` 요청에 `index.html`을 응답하도록 추가**

```js
if (pathname === "/") {
  var indexPath = path.join(__dirname, "index.html");
  fs.readFile(indexPath, "utf8", function (err, html) {
    // 성공하면 index.html 내용을 text/html로 그대로 돌려줌
  });
  return;
}
```

이제 누군가 서버 주소(`/`)로 접속하면, 서버가 같은 폴더에 있는 `index.html` 파일을
읽어서 그대로 화면에 보여줍니다. 기존 API 주소 4개(`/api/trends`, `/api/naver-ready`,
`/api/search-trend-test`, `/api/shopping-insight-test`)는 이전과 완전히 동일하게
동작합니다.

**2) `index.html`: API 서버 주소를 여는 방식에 맞게 자동으로 선택**

이전에는 `http://localhost:3000/api/trends` 주소가 코드에 그대로 고정되어 있어서,
Render에 배포된 화면에서는 항상 내 컴퓨터의 localhost를 찾다가 실패했습니다. 이제는
아래처럼 지금 이 페이지가 어떻게 열렸는지 확인해서 주소를 자동으로 고릅니다.

```js
function getApiBaseUrl() {
  if (window.location.protocol === "file:") {
    // 컴퓨터에서 index.html 파일을 더블클릭해서 열었을 때
    return "http://localhost:3000/api/trends";
  }
  // Render처럼 웹 서버가 index.html까지 함께 내려주는 경우
  return "/api/trends";
}
```

- **로컬에서 `index.html`을 더블클릭해서 열 때**(주소창에 `file://...`로 시작):
  지금까지와 똑같이 `http://localhost:3000/api/trends`로 요청합니다. `node server.js`를
  먼저 실행해둬야 하는 것도 그대로입니다.
- **Render 주소(`https://...`)로 접속했을 때**: `/api/trends`라는 "상대 주소"로
  요청합니다. 상대 주소는 지금 보고 있는 페이지와 같은 서버로 자동 연결되므로,
  Render 주소를 코드에 직접 적을 필요가 없습니다.

### 브라우저에서 확인할 항목 (초보자용)

배포 후 아래 3개 주소를 각각 열어서 확인해 주세요.

1. **`https://trend-top10.onrender.com`**
   "Not Found" 대신 "트렌드 TOP 10" 화면(제목, 키워드 입력 영역 등)이 정상적으로
   보이는지 확인합니다. 페이지가 열리자마자 자동으로 트렌드 조회가 실행되어 표/차트에
   데이터가 채워지는지도 함께 확인합니다.
2. **`https://trend-top10.onrender.com/api/naver-ready`**
   `{"shoppingInsightReady":true,"searchTrendReady":true}`처럼 JSON 응답이 그대로
   보이는지 확인합니다(이전과 동일하게 동작해야 합니다).
3. **`https://trend-top10.onrender.com/api/trends`**
   `searchData`, `shoppingProductData` 등이 담긴 JSON 응답이 보이는지 확인합니다

로컬 확인도 잊지 마세요: `node server.js`를 실행한 뒤 `index.html`을 더블클릭해서

## v2.4 — 운영 점검용 health API 추가

이번 버전은 화면 기능이 아니라, **배포된 서버가 정상적으로 떠 있는지 빠르게 확인할 수

### `server.js`에 추가된 것

**`GET /api/health`** 라우트를 새로 추가했습니다. 네이버 API를 전혀 호출하지 않고,
이 서버 프로세스가 요청을 받아 응답할 수 있는 상태인지만 바로 알려줍니다.

```json
{
  "ok": true,
  "service": "trend-top10",
  "status": "running",
  "timestamp": "2026-07-09T08:09:44.751Z"
}
```

  (`NAVER_...` 관련 값을 하나도 읽지 않고, 고정된 문구와 현재 시각만 돌려줍니다.)
- 네이버 서버에 요청을 보내지 않으므로, 네이버 API 키가 아예 설정되어 있지 않거나
  네이버 서버가 느려도 `/api/health`는 항상 빠르게 응답합니다.

### 기존 라우트는 모두 그대로 유지

- "여러 후보 키워드 중 관심도가 높은 순서"를 보여주는 것보다, **네이버 쇼핑 검색
  API가 실제로 돌려주는 검색 결과 자체**를 보고 싶다는 요구에 맞췄습니다.
- 다만 검색 결과 순서(정확도순)가 구매순위·판매순위·공식 인기순위로 오해되지
  않도록, 화면 곳곳의 안내 문구를 이 버전에서 다시 명확히 정리했습니다.

### `server.js`에 추가된 것

**`GET /api/naver-shopping-search?query=검색어`** 라우트를 새로 추가했습니다.

- 네이버 쇼핑 검색 API(`https://openapi.naver.com/v1/search/shop.json`)를 **서버에서만**
  호출합니다. `display=10`, `start=1`, `sort=sim`(정확도순)을 기본값으로 사용합니다.
- `query`는 필수이며, 비어 있거나 20자를 넘으면 초보자가 이해할 수 있는 `400` 오류
  JSON을 돌려줍니다(예: `{"status":400,"message":"검색어(query)가 필요합니다.", ...}`).
- 네이버 API 호출 자체가 실패하면(키 미설정, 네이버 서버 오류 등) 자세한 상태 코드와
  메시지를 담은 오류 JSON을 그대로 돌려줍니다. **이전 버전(`/api/trends`)과 달리 이
  라우트는 실패를 예시 데이터로 조용히 대체하지 않고, 오류 자체를 알려줍니다.** 예시
  데이터로 보여줄지 말지는 화면(`index.html`) 쪽에서 판단합니다.
- 응답에는 화면에서 바로 쓰기 쉽게 정리한 검색 결과 10개(`items`)와 함께 아래 메타
  정보가 담깁니다.
  ```json
  {
    "source": "naver-shopping-search-api",
    "query": "선크림",
    "display": 10,
    "sort": "sim",
    "notice": "네이버 쇼핑 검색 API 결과이며 구매순위/판매순위/공식 인기순위가 아닙니다.",
    "items": [
      { "rank": 1, "title": "...", "mallName": "...", "lprice": "...", "hprice": "...",
        "productType": "...", "brand": "...", "maker": "...", "category1": "...",
        "category2": "...", "category3": "...", "category4": "...", "link": "...", "image": "..." }
    ]
  }
  ```
- 이 API 키(`NAVER_SHOPPING_CLIENT_ID`/`SECRET`)는 이전 버전(쇼핑인사이트)에서 쓰던
  값을 그대로 재사용합니다. **새 환경변수 이름을 추가하지 않았습니다.**
- 응답과 서버 콘솔 로그 어디에도 API 키/Secret이나 다른 환경변수 값을 출력하지
  않습니다(상태 코드와 고정 안내 문구만 출력합니다).
- 기존 라우트(`/`, `/api/trends`, `/api/naver-ready`, `/api/health`,
  `/api/search-trend-test`, `/api/shopping-insight-test`)는 이번 버전에서도 전혀
  수정하지 않았습니다.

> ⚠️ **실제 배포 환경에서 확인이 필요합니다**: 로컬에서 실제 키로 테스트해본 결과,
> 지금 `.env`에 설정된 `NAVER_SHOPPING_CLIENT_ID`/`SECRET`은 네이버 쇼핑 검색 API
> 호출 시 `401 Scope Status Invalid`(인증 실패) 오류를 돌려줬습니다. 이는 코드 문제가
> 아니라, **해당 Client ID/Secret이 등록된 네이버 개발자센터 애플리케이션에 "검색"
> API(쇼핑)가 아직 활성화되어 있지 않기 때문**으로 보입니다(그동안 이 키 쌍은
> "데이터랩 쇼핑인사이트" 용도로만 써왔습니다). 네이버 개발자센터
> (developers.naver.com) → 해당 애플리케이션 → "API 설정"에서 **검색(쇼핑)** API를
> 추가로 활성화하거나, 검색 API가 포함된 새 애플리케이션의 Client ID/Secret을 발급받아
> Render 환경변수의 `NAVER_SHOPPING_CLIENT_ID`/`SECRET` 값을 갱신해야 실제 검색 결과가
> 정상적으로 표시됩니다. 이 설정이 되어 있지 않은 동안에는 화면에 예시 데이터와 노란색
> "API 호출 실패" 안내가 표시되며, 이는 정상적인 fallback 동작입니다.

### `index.html`에 추가/변경된 것

- **화면의 중심 기능**을 "네이버 쇼핑 검색 결과 TOP10"으로 바꿨습니다. 헤더 제목도
  "네이버 쇼핑 검색 결과 TOP10"으로 바뀌었습니다.
- **검색어 입력을 1개로 단순화**했습니다(기본값: "선크림"). [검색 결과 조회] 버튼을
  누르면 `/api/naver-shopping-search?query=검색어`를 호출합니다. 빈 칸으로 조회하면
  기본 검색어(선크림)를 사용합니다. [기본 검색어로 되돌리기] 버튼도 그대로
  있습니다(입력 칸을 기본값으로 되돌리고, 저장해둔 최근 검색어도 지웁니다).
- **결과 표**가 순위 / 이미지 / 상품명 / 쇼핑몰 / 최저가 / 카테고리 / 링크 7개 열로
  바뀌었습니다. 링크를 누르면 실제 네이버 쇼핑 상품 페이지가 새 탭으로 열립니다. 표가
  넓어진 만큼, 좁은 화면에서는 표 안에서만 가로 스크롤되도록 했습니다.
- **결과가 10개보다 적게 내려오면** 실제 내려온 개수만 표시하고, 표 위에 "ℹ️ 현재
  n개 결과 표시 중" 안내가 나타납니다.
- **막대 차트를 가격 비교용으로 변경**했습니다. 막대 길이는 최저가(`lprice`) 기준
  상대 비교값이며, 인기순위나 판매량과는 관련이 없다는 안내를 차트 위에 함께
  표시합니다.
- **카테고리 필터**는 이번 검색 결과 10개 안에 실제로 등장한 카테고리(`category1`)
  기준으로 매번 새로 만들어집니다.
- **"오늘의 요약"** 카드를 검색 결과 개수 / 최저가 상품 / 가장 많이 등장한 카테고리
  3가지로 다시 계산하도록 바꿨습니다(이전의 "급상승 키워드", "신규 진입 항목"은 후보
  키워드 비교 모델에서만 의미가 있던 개념이라 제거했습니다).
- **"조회 기준" 카드**를 검색어 / 정렬 기준 / 데이터 출처 / 결과 개수 4가지로
  바꿨습니다("조회 기간" 카드는 검색 API에는 해당하지 않아 제거했습니다).
- **결과 요약 문장 / 결과 요약 복사**는 이제 현재 검색어와 검색 결과 10개(또는
  카테고리로 필터링된 결과)를 기준으로 문장을 만들고 복사합니다.
- 화면 오해 방지를 위해 "검색 트렌드 TOP10", "쇼핑 관심도 비교", "후보 키워드 비교"
  같은 표현을 모두 지우고, 아래 문구를 안내 배너와 "사용 안내" 섹션에 명확히
  넣었습니다.
  1. 이 결과는 네이버 쇼핑 검색 API가 반환한 검색 결과입니다.
  2. 구매순위, 판매순위, 네이버 공식 인기순위가 아닙니다.
  3. 기본 정렬은 정확도순(sort=sim)입니다.
  4. 개인 참고용 대시보드입니다.
- **최근 검색어 `localStorage` 기억 기능**은 그대로 유지하되, 저장 키를
  `trendTop10.lastSearchQuery`(검색어 문자열 1개)로 바꿨습니다. 이전 버전의
  `trendTop10.lastKeywords`(키워드 배열) 값은 새 코드에서 더 이상 읽지 않지만, 굳이
  지우지 않아도 안전합니다(그냥 쓰이지 않는 값으로 남을 뿐입니다). 네이버 API 응답
  데이터는 이번에도 어디에도 저장하지 않습니다.
- **"사용 안내"와 "운영 점검 링크" 섹션은 그대로 유지**했습니다. "사용 안내" 목록에
  위 4가지 문구를 추가하고, API 실패 시 예시 데이터 안내와 Render 첫 로딩 지연 안내도
  함께 남겨뒀습니다.

### 안내 배너 구조를 3단계로 정리

결과 목록이 하나로 줄어들면서, 배너 상태를 아래처럼 다시 정리했습니다.

1. **입력 오류**(검색어가 비어있거나 20자 초과, `status 400`): 입력 칸 아래 빨간
   글씨로 표시됩니다.
2. **네이버 API 호출 실패**(검색어는 정상이지만 네이버 쪽에서 오류가 난 경우, `status
   400`이 아닌 오류): 페이지 상단에 노란 배너로 "네이버 쇼핑 검색 API 호출에
   실패했습니다..."가 표시됩니다. (이전 버전의 "일부 목록만 예시 데이터"라는 노란
   배너 문구는, 표가 하나로 줄면서 더 이상 맞지 않아 이번에 문구를 새로 정리했습니다.)
3. **서버 연결 실패**(서버 자체가 꺼져 있거나 네트워크 문제): 페이지 상단에 빨간
   배너로 표시됩니다(문구는 이전과 동일).

### 기존 기능은 모두 그대로 유지

키워드(검색어) 조회, 기본 검색어로 되돌리기, 최근 검색어 기억, 카테고리 필터, 오늘의
요약, 조회 기준 카드, 마지막 조회 시각, 결과 요약 문장, 결과 요약 복사, 표, 차트,
fallback(API 실패 시 예시 데이터 표시), 조회 중/오류 UX, 사용 안내 섹션, 운영 점검
링크는 모두 이번 버전에서도 남아있습니다. 다만 "후보 키워드 비교"라는 기존 모델
자체가 "실제 검색 결과 확인" 모델로 바뀌었기 때문에, 각 기능이 다루는 데이터의
모양(2개 목록 → 1개 목록, 관심도/변동 → 가격/카테고리)은 이번 버전에서 함께
바뀌었습니다.

### 브라우저에서 확인할 항목 (초보자용)

1. 터미널에서 서버를 실행합니다(로컬 확인 시).
   ```
   node server.js
   ```
2. https://trend-top10.onrender.com/ (또는 로컬 `index.html`)에 접속합니다. 헤더
   제목이 "네이버 쇼핑 검색 결과 TOP10"으로 보이는지 확인합니다.
3. 검색어 입력 칸에 원하는 검색어(예: "무선이어폰")를 입력하고 [검색 결과 조회]를
   눌러서, 표와 차트가 그 검색어 기준으로 바뀌는지 확인합니다.
4. 표 제목 옆 배지가 "네이버 쇼핑 검색 API 데이터"(초록)인지 "예시 데이터 표시
   중"(빨강)인지 확인합니다. 위 "실제 배포 환경에서 확인이 필요합니다" 안내에 따라
   네이버 API 쪽 권한 설정이 끝나지 않았다면 예시 데이터와 노란 "API 호출 실패"
   배너가 보이는 것이 정상입니다.
5. "카테고리 필터" 버튼이 이번 검색 결과에 실제로 등장한 카테고리로 채워지는지, 버튼을
   누르면 표/차트/결과 요약 문장이 함께 걸러지는지 확인합니다.
6. 검색 결과가 10개 미만으로 내려오는 검색어를 시도해보고(드문 상품일수록 가능성이
   높습니다), 표 위에 "ℹ️ 현재 n개 결과 표시 중" 안내가 뜨는지 확인합니다.
7. 표의 링크 열을 눌러서 실제 네이버 쇼핑 상품 페이지가 새 탭으로 열리는지 확인합니다.
8. [결과 요약 복사]를 눌러 메모장 등에 붙여넣고, 검색어/정렬 기준/결과 10개/"구매순위
   등이 아니다" 안내 문구가 모두 포함되어 있는지 확인합니다.
9. 화면 맨 아래 "4. 사용 안내"에 위 4가지 새 문구가 포함되어 있는지, "운영 점검 링크"
   3개가 그대로 남아 있는지 확인합니다.
10. 검색어 입력 칸을 비운 채 [검색 결과 조회]를 눌러서 기본 검색어(선크림)로 조회되는지,
    20자가 넘는 검색어를 입력해서 빨간 "입력 오류" 문구가 뜨는지 확인합니다.

### 화면에 추가/변경된 것

- **가격 요약 카드**를 결과 표 위쪽에 새로 추가했습니다. 최저가 / 최고가 / 평균가 /

### `server.js`에 추가된 것

**`GET /api/market-status`** 라우트를 새로 추가했습니다.

- 미리 정해둔 관심 키워드 10개(선크림, 양산, 여름원피스, 쿨토시, 제습기, 미니선풍기,
  래쉬가드, 샌들, 냉감이불, 물놀이용품)의 최근 30일 검색어트렌드·쇼핑인사이트 추이를
  네이버 데이터랩 API로 조회합니다.
- 네이버 데이터랩 API는 **한 번에 최대 5개까지만** 키워드(그룹)를 비교할 수 있어서,
  10개를 5개씩 두 묶음으로 나눠 검색어트렌드·쇼핑인사이트 각각 두 번씩, 총 4번
  호출한 뒤 결과를 합칩니다.
- 응답에는 기준 기간(`dateRange`), 관심 키워드 목록(`keywords`), 키워드별 관심도
  점수·상승/하락 여부(`keywordData`), 데이터 출처(`source: "naver-datalab"`), 그리고
  안내 문구(`notice`)가 담깁니다.
  ```json
  {
    "source": "naver-datalab",
    "notice": "네이버 공식 인기검색어 순위가 아니라 관심 키워드 기준 참고용 시장현황입니다.",
    "dateRange": { "startDate": "2026-06-09", "endDate": "2026-07-08" },
    "keywords": ["선크림", "양산", "..."],
    "keywordData": [
      { "keyword": "선크림", "category": "뷰티", "searchInterestScore": 62,
        "searchTrend": "up", "searchTrendDiff": 6, "shoppingClickScore": 40 }
    ],
    "partialFailure": false
  }
  ```
- 검색어트렌드 API가 두 묶음 모두 실패해서 쓸만한 데이터가 하나도 없으면, 화면
  쪽에서 예시 데이터로 대체할 수 있도록 오류(`502`)로 응답합니다. 두 묶음 중
  하나만 실패한 경우에는 나머지 데이터를 그대로 내려주되 `partialFailure: true`로
  표시해서, 화면이 "일부 데이터를 가져오지 못했다"는 안내를 보여줄 수 있게 했습니다.
- 이 라우트도 기존 키(`NAVER_SEARCH_TREND_CLIENT_ID`/`SECRET`,
  `NAVER_SHOPPING_CLIENT_ID`/`SECRET`)를 그대로 재사용합니다. **새 환경변수 이름을
  추가하지 않았습니다.**
- 응답과 서버 콘솔 로그 어디에도 API 키/Secret이나 다른 환경변수 값을 출력하지
  않습니다.
- 기존 라우트(`/`, `/api/trends`, `/api/naver-ready`, `/api/health`,
  `/api/naver-shopping-search`, `/api/search-trend-test`,
  `/api/shopping-insight-test`)는 이번 버전에서도 전혀 수정하지 않았습니다.

> ℹ️ **관심도 점수에 대한 참고**: 네이버 데이터랩 API는 한 번의 요청 안에서 비교한
> 키워드끼리만 0~100으로 정규화된 값을 돌려줍니다. 관심 키워드 10개를 5개씩 두 번
> 나눠 요청하기 때문에, 서로 다른 묶음에 속한 키워드끼리의 "관심도 점수" 절대값
> 비교는 근사치입니다(예: 1번 묶음의 "40점"과 2번 묶음의 "40점"이 정확히 같은
> 관심도라는 뜻은 아닙니다). 반면 **한 키워드 안에서의 상승/하락 여부**는 같은 요청
> 안에서 계산되므로 그대로 신뢰할 수 있습니다. 이 화면은 애초에 "참고용 개인
> 대시보드"이므로 이 근사치를 그대로 사용하되, 공식 순위처럼 소개하지 않습니다.

## v2.9 — 상세 결과 표를 가격 비교 중심으로 개선(가격순위 추가, 차트 제거)

이번 버전은 새 API가 아니라, "3. 상세 결과" 표를 **가격을 비교해서 실제로 저렴한
상품을 바로 찾을 수 있도록** 다듬는 것이 목표입니다. `server.js`는 이번 버전에서
전혀 수정하지 않았고, `.env` 파일도 건드리지 않았습니다. `index.html`에는 이번에도
API 키/Secret을 전혀 추가하지 않았습니다. 네이버 API 응답 원본은 이번에도 어디에도
저장하지 않으며, `localStorage`에는 최근 검색어/시장현황 관심 키워드 텍스트만
저장됩니다(둘 다 이전 버전과 동일).

### 검색순위와 가격순위는 어떻게 다른가요? (초보자용)

표에는 이제 서로 다른 의미를 가진 순위 2개가 함께 표시됩니다.

- **검색순위**: 네이버 쇼핑 검색 API가 응답으로 돌려준 순서 그대로입니다(이전
  버전까지 "순위"라고만 부르던 열의 새 이름입니다). 기본 정렬 기준은 정확도순
  (`sort=sim`)이며, **네이버 공식 인기순위나 판매순위가 아닙니다.**
- **가격순위**: **지금 화면에 보이는 결과 안에서만** 최저가(`lprice`) 기준으로 다시
  매긴 참고용 순위입니다. 가격 정보가 없는 상품은 순위 계산에서 빠지고 "-"로
  표시됩니다. 카테고리 필터를 바꾸면 "지금 보이는 결과"가 바뀌므로, 가격순위도 그
  기준으로 다시 계산됩니다 — 예를 들어 전체 결과에서는 가격순위 5위였던 상품이,
  특정 카테고리만 골라보면 그 안에서는 가격순위 1위가 될 수 있습니다.

두 순위 모두 **네이버 공식 구매순위·판매순위·인기순위가 아닌 참고용 정보**입니다.

### `index.html`에 변경된 것

- 표의 **기존 "순위" 열 이름을 "검색순위"로 바꿨습니다.**
- **"최저가" 열 바로 오른쪽에 "가격순위" 열을 새로 추가**했습니다. 표 열 순서는
  다음과 같습니다: 검색순위 / 이미지 / 상품명 / 쇼핑몰 / 최저가 / 가격순위 /
  카테고리 / 상품 보기.
- **가격순위 1~3위는 파란색 배지로 눈에 띄게 표시**됩니다(1위가 가장 진한 파란색,
  2·3위는 옅어짐). 4위 이후는 회색 배지, 가격 정보가 없는 상품은 "-"로 표시됩니다.
  검색순위(파란 굵은 숫자, 행 배경색 강조)와 헷갈리지 않도록 서로 다른 색과 모양을
  사용했습니다.
- **가격 비교 막대 차트를 제거**했습니다. 막대 길이 비교보다 표 안의 가격순위
  배지가 "지금 결과 안에서 뭐가 저렴한지"를 더 바로 알려준다고 판단했습니다. 표와
  가격 요약 카드(최저가/최고가/평균가/결과 개수) 중심으로 정리했습니다. 가격 요약
  카드는 이전 버전과 동일하게 유지됩니다.
- **결과 요약 문장**에 가격순위 1위 상품을 명시하는 문장이 추가됐습니다. 예:
  "현재 표시 결과 중 최저가는 '○○'(12,800원)이며, 가격순위 1위입니다."
- **[결과 요약 복사]** 텍스트의 상품 목록에도 각 상품의 가격순위가 함께
  담깁니다(예: "검색순위 3위 ○○ (몰이름) 12,800원 [가격순위 1위] [화장품/미용]").
- 표 아래 안내 문구와 "사용 안내" 목록에 검색순위/가격순위의 의미와 "구매순위·
  판매순위·공식 인기순위가 아니다"라는 안내를 추가로 명시했습니다.
- 모바일 화면에서도 검색순위·최저가·가격순위·[상품 보기] 버튼이 겹치지 않고 보기
  좋게 표시되도록 배지 크기를 줄이는 CSS를 추가했습니다.
