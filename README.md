# 트렌드 TOP 10

"트렌드 TOP 10"은 **쇼핑 검색 상품 TOP 10**과 **검색 트렌드 TOP 10**을 보여주는
대시보드형 홈페이지입니다. 코딩 초보자가 Claude Code와 함께 만들고 있는 연습 프로젝트이며,
빌드 도구나 외부 라이브러리 없이 `index.html` 파일 하나로 동작합니다. 화면에는 네이버 API로
직접 조회한 데이터 또는(호출 실패 시) 예시(샘플) 데이터가 표시될 수 있습니다.

## 실행 방법

이제 `index.html`은 화면에 보여줄 데이터를 `server.js`(가짜 API 서버)에게 요청해서
받아옵니다. 그래서 아래 순서로 실행해야 정상적으로 데이터가 보입니다.

1. 이 폴더에서 터미널(PowerShell)을 열고 아래 명령어로 서버를 먼저 실행합니다.
   ```
   node server.js
   ```
   `가짜 API 서버 실행 중: http://localhost:3000/api/trends` 메시지가 보이면 성공입니다.
2. 서버를 켜둔 상태로 `index.html` 파일을 더블클릭해서 브라우저로 엽니다.

만약 서버를 켜지 않고 `index.html`만 열면, 데이터를 받아오지 못했다는 안내 배너가
화면에 표시됩니다.

## 현재 버전: v0.6.1

지금까지 완성된 기능은 다음과 같습니다.

- **상단 제목 영역**: "트렌드 TOP 10" 제목과 부제, 오늘 날짜(기준 날짜)를 자동으로 표시
- **데이터 안내 배너**: 네이버 API 기반 데이터 또는 예시(샘플) 데이터가 표시될 수 있음을
  헤더 바로 아래 눈에 띄는 배너로 안내
- **오늘의 트렌드 요약**: 아래 3가지 값을 고정 문구가 아니라 데이터에서 직접 계산해서 표시
  - 가장 핫한 카테고리: 전체 데이터 중 가장 많이 등장하는 카테고리
  - 급상승 키워드: 검색 트렌드 데이터 중 순위가 가장 많이 오른 항목
  - 신규 진입 항목: 순위가 새로 등장(`new`)한 항목 개수
- **카테고리 선택 버튼**: 전체 / 패션의류 / 뷰티 / 생활용품 / 식품 / 디지털가전 / 스포츠레저
  버튼을 눌러 두 표를 동시에 필터링
- **쇼핑 검색 상품 TOP 10 / 검색 트렌드 TOP 10 표**: 순위 / 항목 / 카테고리 / 변동 4개 열로 구성
  - 변동 열은 상승(▲) / 하락(▼) / 변동없음 / 신규 진입(NEW)을 구분해서 표시
  - 1위·2위·3위 행은 배경색으로 강조 (1위가 가장 눈에 띄게, 2·3위는 약하게)
- **모바일 반응형**: 화면이 좁아지면 두 표가 가로가 아니라 세로로 쌓이도록 처리
- **API 서버 (`server.js`)**: `index.html`이 고정된 배열 대신 서버에 요청해서 데이터를
  받아오도록 구조를 바꿨습니다 (`GET /api/trends`). 서버가 꺼져 있으면 화면에 안내
  배너가 표시됩니다.
- **네이버 API 연동 준비 (용도별 키 분리)**: `server.js`가 **쇼핑인사이트(쇼핑 검색
  상품용)**와 **검색어트렌드(검색 트렌드용)** API 키를 각각 따로 읽어오는 구조를
  만들어뒀습니다.
  - `NAVER_SHOPPING_CLIENT_ID` / `NAVER_SHOPPING_CLIENT_SECRET`: 쇼핑 검색 상품(쇼핑인사이트)용
  - `NAVER_SEARCH_TREND_CLIENT_ID` / `NAVER_SEARCH_TREND_CLIENT_SECRET`: 검색 트렌드(검색어트렌드)용
  - 위 값이 없으면 기존 `NAVER_CLIENT_ID` / `NAVER_CLIENT_SECRET`을 공용으로 사용합니다(fallback).
  - `GET /api/naver-ready`로 두 API가 각각 준비됐는지 확인할 수 있습니다.
    ```json
    { "shoppingInsightReady": true, "searchTrendReady": false }
    ```

## 네이버 API 키 발급 및 환경변수 설정 방법 (초보자용)

아직 이 프로젝트는 실제로 네이버 API를 호출하지 않지만, 나중에 연동할 때를 대비해서
키를 미리 발급받고 환경변수로 설정하는 방법을 정리해뒀습니다.

네이버 API는 기능(상품)별로 따로 신청하기 때문에, **구매 순위(쇼핑인사이트)**와
**검색 순위(검색어트렌드)**를 각각 신청하면 Client ID/Secret이 서로 다른 키 "두 쌍"으로
나올 수도 있고, 신청 방식에 따라 하나의 애플리케이션에 두 API를 함께 등록해서 키 "한 쌍"만
받을 수도 있습니다. 두 경우 모두 지원하도록 만들어뒀으니, 아래에서 내 상황에 맞는 방법을
따라 하면 됩니다.

### 1. 네이버 개발자센터에서 API 키 발급받기

1. 네이버 개발자센터(developers.naver.com)에 접속해서 네이버 계정으로 로그인합니다.
2. "Application" 메뉴에서 애플리케이션을 새로 등록합니다.
3. 사용할 API를 선택합니다.
   - 구매 순위용: **쇼핑인사이트** API
   - 검색 순위용: **검색어트렌드** API
   - 둘 다 필요하면 신청 화면에서 두 API를 모두 체크해도 되고, 따로따로 애플리케이션을
     만들어도 됩니다.
4. 등록이 끝나면 **Client ID**와 **Client Secret** 값을 발급받습니다.
   - 이 값들은 비밀번호와 같습니다. 다른 사람에게 공유하거나 코드에 직접 적지 마세요.

### 2. `.env` 파일 채우기 — 키가 한 쌍인 경우

하나의 애플리케이션에서 두 API를 함께 등록해서 **Client ID/Secret을 한 쌍만** 받았다면,
`.env.example`에서 아래 두 줄의 주석(`#`)만 지우고 값을 채우면 됩니다. 나머지 4개 줄은
그대로 주석 처리된 상태로 두거나 지워도 괜찮습니다.

```
NAVER_CLIENT_ID=발급받은_클라이언트_ID
NAVER_CLIENT_SECRET=발급받은_클라이언트_시크릿
```

`server.js`는 용도별 환경변수(`NAVER_SHOPPING_...`, `NAVER_SEARCH_TREND_...`)가 없으면
이 공용 키를 구매/검색 두 곳 모두에 자동으로 사용합니다.

### 3. `.env` 파일 채우기 — 키가 두 쌍인 경우

구매 순위용과 검색 순위용 키를 **따로** 발급받았다면, 용도에 맞게 아래 4개 값을 각각
채워 넣습니다.

```
NAVER_SHOPPING_CLIENT_ID=쇼핑인사이트_클라이언트_ID
NAVER_SHOPPING_CLIENT_SECRET=쇼핑인사이트_클라이언트_시크릿

NAVER_SEARCH_TREND_CLIENT_ID=검색어트렌드_클라이언트_ID
NAVER_SEARCH_TREND_CLIENT_SECRET=검색어트렌드_클라이언트_시크릿
```

이렇게 용도별로 값을 채우면, 위의 공용 `NAVER_CLIENT_ID` / `NAVER_CLIENT_SECRET`이 있어도
용도별 값이 우선 사용됩니다.

### 4. 공통: `.env` 파일 준비 및 주의사항

1. 아직 `.env` 파일이 없다면, 이 프로젝트 폴더에 있는 `.env.example` 파일을 복사해서
   이름을 `.env`로 바꿉니다.
2. 위 2번 또는 3번 방법으로 값을 채워 넣고 저장합니다.
3. `.env` 파일은 절대 다른 사람과 공유하거나 깃허브 같은 곳에 올리지 마세요. 이 파일은
   내 컴퓨터에만 있어야 하는 비밀 파일입니다.

`server.js`는 실행될 때 `.env` 파일이 있으면 자동으로 읽어서 환경변수로 등록합니다
(별도의 외부 라이브러리 설치 없이 동작합니다).

### 5. 환경변수가 잘 설정됐는지 확인하기

1. 터미널에서 서버를 실행합니다.
   ```
   node server.js
   ```
2. 브라우저나 터미널에서 `http://localhost:3000/api/naver-ready` 주소에 접속합니다.
3. 아래처럼 두 값이 각각 표시됩니다.
   ```json
   { "shoppingInsightReady": true, "searchTrendReady": true }
   ```
   - `shoppingInsightReady`가 `true`면 구매 순위(쇼핑인사이트)용 키가 준비된 것입니다.
   - `searchTrendReady`가 `true`면 검색 순위(검색어트렌드)용 키가 준비된 것입니다.
   - `false`가 보이면 해당 용도의 `.env` 값(또는 공용 `NAVER_CLIENT_ID`/`SECRET`)을
     다시 확인해야 합니다.

> ⚠️ **다시 한번 강조**: 네이버 API 키(Client ID)나 Secret 값은 절대 `index.html`
> (또는 다른 클라이언트/브라우저 코드)에 직접 적지 마세요. `index.html`은 누구나
> "페이지 소스 보기"로 내용을 볼 수 있기 때문에, 여기에 키를 적으면 그 즉시 유출됩니다.
> API 키와 Secret은 반드시 서버 쪽 환경변수(`.env` 파일)로만 보관해야 합니다.

## v0.7 — 실제 네이버 API 호출 테스트

`index.html`은 이번 버전에서 **아직 수정하지 않았습니다.** 화면이 실제로 사용하는
`/api/trends`도 여전히 예전처럼 예시(가짜) 데이터를 그대로 돌려줍니다.

이번 버전에서는 `server.js`에 "테스트 전용" 주소 2개를 추가해서, 네이버 API 키가
잘 작동하는지 미리 확인만 해볼 수 있게 했습니다. 두 주소 모두 서버(`server.js`)가
네이버 서버에 직접 요청을 보내고, 받아온 응답을 거의 그대로 화면에 보여줍니다.

### 테스트 전 준비

1. `.env` 파일에 `NAVER_SEARCH_TREND_CLIENT_ID/SECRET`(검색어트렌드용) 또는
   `NAVER_SHOPPING_CLIENT_ID/SECRET`(쇼핑인사이트용) 값이 채워져 있어야 합니다.
   (둘 다 없으면 공용 `NAVER_CLIENT_ID`/`NAVER_CLIENT_SECRET`을 대신 사용합니다.)
   자세한 설정 방법은 위 "네이버 API 키 발급 및 환경변수 설정 방법" 절을 참고하세요.
2. 터미널에서 서버를 실행합니다.
   ```
   node server.js
   ```

### 1. 검색어트렌드 테스트: `/api/search-trend-test`

브라우저 주소창에 아래 주소를 입력합니다.

```
http://localhost:3000/api/search-trend-test
```

- "선크림", "양산", "여름원피스" 3개 검색어의 최근 30일(어제까지) 검색 관심도 추이를
  네이버 데이터랩 검색어트렌드 API로 직접 조회한 결과를 그대로 보여줍니다.
- 정상적으로 동작하면 각 검색어별로 날짜(`period`)와 상대적 검색 비율(`ratio`)이 담긴
  JSON이 보입니다.

### 2. 쇼핑인사이트 테스트: `/api/shopping-insight-test`

브라우저 주소창에 아래 주소를 입력합니다.

```
http://localhost:3000/api/shopping-insight-test
```

- 카테고리 코드 `50000000`(문서 예시 값) 안에서 "선크림" 키워드의 최근 30일 쇼핑 클릭
  추이를 네이버 데이터랩 쇼핑인사이트 API로 직접 조회한 결과를 그대로 보여줍니다.
- 카테고리 코드가 실제로 "선크림"과 맞물리는 분야가 아니면 `data`가 빈 배열(`[]`)로
  올 수도 있는데, 이는 API 호출 자체는 성공했다는 뜻이니 정상입니다. (나중에 실제
  연동할 때 카테고리 코드를 우리 데이터에 맞는 값으로 바꾸면 됩니다.)

### 오류가 날 때

두 주소 모두 네이버 API 호출이 실패하면(키가 없거나, 키가 틀렸거나, 네이버 서버 오류
등) 아래처럼 `status`, `message`, `detail` 3개 값을 담은 JSON을 보여줍니다. 실제 API
키나 Secret 값은 어떤 경우에도 응답이나 서버 콘솔에 출력되지 않습니다.

```json
{
  "status": 401,
  "message": "네이버 API가 오류를 반환했습니다.",
  "detail": { "errorMessage": "..." }
}
```

## v0.8 — 네이버 API 응답을 화면 데이터에 연결

`index.html`이 실제로 데이터를 요청하는 `GET /api/trends`가 이제 네이버 API를 직접
호출합니다. `index.html` 자체는 이번에도 API 키/Secret을 전혀 담고 있지 않습니다
(키는 여전히 `server.js`와 `.env`에만 있습니다). 대신 아래처럼 화면에 보이는 문구를
사실과 다르지 않게 바꿨습니다.

- "구매 순위 TOP 10" → **"쇼핑 검색 상품 TOP 10"**
- "검색 순위 TOP 10" → **"검색 트렌드 TOP 10"**
- 예시 데이터 배너 → **"네이버 API 기반 데이터 또는 예시 데이터가 표시될 수 있습니다"**로 변경

### `/api/trends`가 하는 일

1. v0.7 테스트 때 썼던 것과 같은 키워드("선크림", "양산", "여름원피스")로 네이버
   검색어트렌드 API와 쇼핑인사이트 API를 각각 호출합니다.
2. 호출이 성공하면, 응답에 담긴 관심도 수치(`ratio`)를 이용해 키워드를 순위(rank)로
   정렬하고, 화면 표가 쓰는 `{ rank, name, category, change }` 모양으로 바꿔서
   돌려줍니다.
   - `searchData`: 검색어트렌드 API 결과로 만든 "검색 트렌드 TOP 10" 표 데이터
   - `shoppingProductData`: 쇼핑인사이트 API 결과로 만든 "쇼핑 검색 상품 TOP 10" 표 데이터
3. 호출이 실패하거나(키가 없거나 틀림, 네이버 서버 오류 등) 응답에 쓸만한 데이터가
   없으면, 그 목록만 기존 예시(가짜) 데이터로 대신 채웁니다. 검색/쇼핑 중 한쪽만
   실패해도 나머지 한쪽은 실제 데이터를 그대로 보여줍니다.
4. 어떤 경우에도 네이버 API 응답이나 예시 데이터를 파일이나 DB에 저장하지 않고,
   요청이 들어올 때마다 새로 계산해서 그때그때 돌려줍니다.

### ⚠️ 주의사항: API 응답을 왜곡하지 않기

- 표에 보이는 **순위(rank)**는 네이버가 공식적으로 발표하는 "실시간 쇼핑/검색 순위"가
  아니라, 우리가 미리 정해둔 테스트 키워드 몇 개만 놓고 관심도(`ratio`) 평균이 높은
  순서로 매긴 것입니다. 키워드 종류를 늘리거나 바꾸면 순위도 달라집니다.
- **변동(change)** 값도 네이버가 제공하는 값이 아니라, 최근 7일 평균과 이전 7일 평균의
  관심도 차이를 서버가 직접 계산한 근사치입니다(`server.js`의 `computeChangeFromSeries`
  함수). "선크림이 실제로 검색 3위다" 같은 식으로 공식 통계처럼 소개하지 않도록
  주의해야 합니다.
- 카테고리 값(뷰티/패션의류/생활용품 등)도 네이버 응답에 들어있는 값이 아니라, 테스트
  키워드 각각에 대해 코드에서 미리 정해둔 매핑(`TREND_TEST_KEYWORD_CATEGORY`)입니다.
  키워드를 추가할 때는 이 매핑도 함께 채워줘야 합니다.
- 네이버 원본 응답의 숫자(`ratio` 등) 자체는 가공하지 않고 그대로 계산에만 사용하며,
  임의로 수치를 부풀리거나 순서를 조작하지 않습니다.

### `/api/trends` 응답에 추가된 값

```json
{
  "searchData": [...],
  "searchDataSource": "naver",
  "shoppingProductData": [...],
  "shoppingProductSource": "naver",
  "categories": [...],
  "baseDate": "...",
  "activeKeywords": ["선크림", "양산", "여름원피스"],
  "dateRange": { "startDate": "2026-06-06", "endDate": "2026-07-05" }

복사되는 텍스트는 대략 아래와 같은 모양입니다.

```

검색 트렌드 TOP 10 (카테고리: 전체)
1위 선크림 (뷰티) ▲5
...

순위 산정 방식: 관심도(ratio) 평균 기준 참고 순위 (네이버 공식 순위 아님)

## v1.4 — 최근 조회 키워드 기억 기능

이제 마지막으로 조회했던 키워드를 브라우저가 기억해뒀다가, 페이지를 다시 열었을 때
입력 칸에 자동으로 채워줍니다. `server.js`는 이번 버전에서 전혀 수정하지 않았고,
`index.html`에도 API 키/Secret은 이번에도 추가하지 않았습니다.

### `localStorage`가 무엇인가요? (초보자용)

`localStorage`는 **브라우저가 내 컴퓨터(정확히는 지금 쓰고 있는 그 브라우저)에만 저장해두는
작은 저장 공간**입니다. 서버나 다른 사람의 컴퓨터로 전송되지 않고, 오직 "이 브라우저에서
이 사이트를 다시 열었을 때" 꺼내볼 수 있는 값입니다. 브라우저 저장소를 직접 지우거나
시크릿/프라이빗 모드로 열면 값이 남아있지 않을 수 있습니다.

이 프로젝트에서는 `localStorage`에 **사용자가 입력한 키워드 텍스트**만 저장합니다.
네이버 API가 돌려준 순위·변동 같은 실제 데이터는 이 기능과 전혀 관계없이 이전 버전부터

### 조회 중 표시 개선

- [트렌드 조회] 버튼을 누르면 버튼 문구가 **"조회 중..."**으로 바뀌고, 응답이 올 때까지
  다시 눌리지 않도록 비활성화됩니다. (버튼이 원래 상태로 돌아오는 시점은 이전과 동일하게
  성공/실패와 상관없이 응답을 받은 직후입니다.)
- 같은 시간 동안 **순위 표 영역("쇼핑 검색 상품 TOP 10" / "검색 트렌드 TOP 10")**과
  **"결과 요약 문장" 영역**이 살짝 흐리게(블러) 보여서, "지금 화면 내용이 바뀌는 중"임을
  시각적으로 알 수 있습니다.
- 조회가 끝나면(성공이든 실패든) 흐림 효과가 사라지고 원래대로 선명하게 돌아옵니다.

### 안내 문구를 세 가지 상태로 명확히 구분

이전에는 "서버 연결 실패"와 "일부 데이터가 예시 데이터로 대체된 상태"가 잘 구분되지
않았습니다. v1.7부터는 아래 3가지 상태를 서로 다른 색과 문구로 구분합니다.

1. **입력 오류** (키워드 형식이 잘못된 경우): 입력 칸 아래에 빨간 글씨로
   **"⚠️ 입력 오류: ..."**가 표시됩니다. (기존과 같은 위치, 문구 맨 앞에 "입력 오류"를
   명시해서 다른 두 상태와 헷갈리지 않게 했습니다.)
2. **서버 연결 실패** (서버가 꺼져 있거나 네트워크 문제인 경우): 페이지 상단에 빨간
   배너로 **"⚠️ 서버 연결 실패: 화면에는 이전에 불러온 결과(또는 예시 데이터)가 그대로
   표시되고 있습니다..."**가 표시됩니다.
3. **API 일부 실패(fallback) 상태** (서버 응답 자체는 성공했지만, 네이버 API 호출
   일부가 실패해서 예시 데이터로 대체된 경우): 페이지 상단에 새로 생긴 노란 배너로
   **"ℹ️ 조회는 정상적으로 처리됐지만, 일부 데이터는 네이버 API 호출에 실패해 예시
   데이터로 대체되어 표시 중입니다..."**가 표시됩니다. (표 제목 옆 배지와 함께 보면
   어떤 표가 예시 데이터인지 바로 알 수 있습니다.)

### 실패해도 직전 성공 결과가 화면에 남습니다

입력 오류나 서버 연결 실패가 발생해도, 순위 표·데이터 출처 배지·조회 기준 카드·결과
요약 문장 등은 **직전에 성공했던 조회 결과 그대로** 남아 있습니다. (이 동작은 사실
v1.3~v1.6에서부터 실패 시 데이터를 따로 지우지 않는 방식으로 이미 만들어져 있었고,
v1.7에서 안내 문구를 더 명확히 다듬으면서 그대로 유지했습니다.)

- 페이지를 열자마자 **첫 조회부터 실패**해서 아직 성공한 결과가 하나도 없는 경우에는,
  미리 준비해둔 예시 데이터가 표시되고 표 제목 옆 배지에도 "예시 데이터 표시 중"이라고
  명확히 표시됩니다.
- **마지막 조회 시각**은 이전과 동일하게 조회가 성공했을 때만 갱신되고, 실패 시에는
  그대로 유지됩니다.
- **[결과 요약 복사]**도 이전과 동일하게, 지금 화면에 남아 있는 결과(성공한 마지막
  조회 결과, 또는 예시 데이터)를 기준으로 텍스트를 만듭니다.

### 기존 기능은 모두 그대로 유지

키워드 조회, 기본 키워드로 되돌리기, 최근 조회 키워드 기억, 카테고리 필터, 오늘의
트렌드 요약, 데이터 출처 배지, 조회 기준 카드, 마지막 조회 시각, 결과 요약 문장, 결과
요약 복사, 1~3위 강조, API 호출 실패 시 예시 데이터로 대체(fallback)하는 기능은 이번
버전에서 전혀 바뀌지 않았습니다.

## v1.8 — 간단 막대 차트 기능

이제 표 아래에 순위를 한눈에 비교할 수 있는 막대 차트가 함께 표시됩니다. 외부 차트
라이브러리나 CDN은 전혀 쓰지 않고, 순수 HTML/CSS/JavaScript만으로 만들었습니다.
`server.js`는 이번 버전에서 전혀 수정하지 않았고, `index.html`에도 API 키/Secret은
이번에도 추가하지 않았습니다.

### 화면에 추가된 것

"쇼핑 검색 상품 TOP 10"과 "검색 트렌드 TOP 10" 표 바로 아래에 각각 가로 막대 차트가
새로 생겼습니다. 막대 하나마다 아래 내용이 함께 표시됩니다.

- 순위 (예: "3위")
- 항목 이름
- 변동 표시 (▲ 상승 / ▼ 하락 / 변동없음 / NEW, 표에 쓰는 것과 같은 배지)
- 순위를 기준으로 계산한 상대 길이의 막대

### 막대 길이는 어떻게 계산하나요? (초보자용, 중요)

네이버 API가 실제로 돌려주는 관심도(ratio) 원본 수치는 서버(`server.js`)가 화면
(`index.html`)으로 내려주지 않습니다(순위로 정렬한 뒤 `rank`/`name`/`category`/`change`
값만 내려줍니다). 그래서 이 차트의 막대 길이는 **실제 ratio 수치가 아니라, 화면에 있는
순위(1~10)를 거꾸로 뒤집어 만든 상대 점수**입니다(1위가 가장 길고, 순위가 낮을수록
막대도 짧아집니다). 각 차트 위에 있는 작은 안내 문구
("막대 길이는 순위를 기준으로 계산한 상대 점수이며, 실제 관심도(ratio) 수치가
아닙니다.")가 바로 이 내용을 설명합니다. 순위 자체도 이미 "네이버 공식 구매 순위가
아닌 참고용 계산 결과"라는 점은 이전 버전과 동일합니다.

### 카테고리 필터, fallback과 함께 동작

- 카테고리 버튼을 누르면 표와 차트가 **동시에** 현재 화면 기준으로 다시 그려집니다.
  특정 카테고리를 선택하면 그 카테고리에 해당하는 항목만 차트에 남습니다.
- 네이버 API 호출 일부가 실패해서 예시 데이터가 표시 중이면(v1.7의 노란 "일부 데이터는
  예시 데이터로 대체" 배너 상태), 차트도 그 예시 데이터를 그대로 반영합니다. 표 제목
  옆 데이터 출처 배지는 기존과 동일하게 표 하나에만 붙어 있고, 차트는 별도의 배지 없이
  같은 데이터를 시각화만 합니다.
- 조회 중일 때는(v1.7에서 추가한 블러 효과) 표와 함께 차트 영역도 똑같이 살짝
  흐리게 보였다가, 응답을 받으면 다시 선명해집니다.

### 결과 요약 복사에는 포함되지 않음

[결과 요약 복사]로 복사되는 텍스트는 이전과 동일하게 순수 텍스트만 담습니다. 차트
이미지나 막대 그래프는 복사 텍스트에 포함되지 않습니다.

### 모바일 화면 대응

화면 폭이 좁아지면 차트의 글자 크기와 막대 높이가 살짝 줄어들고, 항목 이름이 길면
말줄임표(...)로 잘려서 가로로 넘치지 않고 화면 안에 깔끔하게 들어맞습니다.

### 기존 기능은 모두 그대로 유지

키워드 조회, 기본 키워드 되돌리기, 최근 조회 키워드 기억, 카테고리 필터, 오늘의 트렌드
요약, 데이터 출처 배지, 조회 기준 카드, 마지막 조회 시각, 결과 요약 문장, 결과 요약
복사, 1~3위 강조, fallback, 조회 중/오류 UX 기능은 이번 버전에서 전혀 바뀌지
않았습니다.

### 브라우저에서 확인할 항목 (초보자용)

1. 터미널에서 서버를 실행합니다.
   ```
   node server.js
   ```
2. 서버를 켜둔 상태로 `index.html`을 더블클릭해서 브라우저로 엽니다.

## v1.9 — 리포트 화면 구조 정리

이번 버전은 새 기능이 아니라, **지금까지 쌓인 여러 영역을 읽기 좋은 순서로 정리하는
것**이 목표입니다. 기능은 하나도 지우거나 바꾸지 않았고, 화면 배치와 제목만 다듬었습니다.
`server.js`는 전혀 수정하지 않았고, `index.html`에도 API 키/Secret이나 외부
라이브러리/CDN은 추가하지 않았습니다.

### 화면을 3단계 큰 그룹으로 재배치

이전에는 "오늘의 트렌드 요약", "카테고리 선택", "관심 키워드 입력", "조회 기준", "결과
요약 문장", "결과 요약 복사", "표/차트"가 순서 없이 나열되어 있었습니다. v1.9부터는
아래 3개의 큰 그룹으로 묶고, 위에서 아래로 **"입력 → 요약 → 상세"** 순서로 읽히도록
정리했습니다. 각 그룹 제목은 파란 밑줄이 있는 굵은 글씨로 표시되어, 지금 화면의 어느
단계를 보고 있는지 한눈에 알 수 있습니다.

## Render 배포 전 최종 점검

`server.js`(가짜 API 서버)를 [Render](https://render.com)에 배포하기 전에 마지막으로
점검한 내용입니다. 실제 화면/기능은 바뀌지 않았고, 배포 환경에 맞춰 포트 처리 방식만
고쳤습니다.

### 점검 항목과 결과

| 점검 항목 | 결과 |
| --- | --- |
| `server.js`가 포트를 `process.env.PORT \|\| 3000` 방식으로 읽는가? | ⚠️ → ✅ 수정함 (기존에는 `server.listen(3000)`으로 고정되어 있었습니다) |
| `index.html`에 API 키/Secret이 있는가? | ✅ 없음 (다시 전체 검사, 이전과 동일하게 안전) |
| `.env`가 `.gitignore`에 의해 Git에서 제외되는가? | ✅ 포함되어 있음 (`.env` 항목이 그대로 있고, `.claude/` 항목도 추가되어 있음을 확인) |

### 실제로 바뀐 부분

`server.js` 맨 아래의 서버 실행 부분을 아래처럼 고쳤습니다.

```js
// 이전 (포트가 3000으로 고정되어 있어 Render에서는 동작하지 않음)
server.listen(3000, function () { ... });

// 이후 (Render가 알려주는 PORT 환경변수가 있으면 그 값을, 없으면 3000을 사용)
var PORT = process.env.PORT || 3000;
server.listen(PORT, function () { ... });
```

Render 같은 호스팅 서비스는 서버가 실제로 어떤 포트 번호를 써야 하는지 자기들이
결정한 뒤 `PORT`라는 환경변수로 알려줍니다. 포트 번호가 3000으로 고정되어 있으면
Render 위에서 서버가 아예 뜨지 않을 수 있어서, 이번에 `process.env.PORT || 3000`
방식으로 바꿨습니다. 내 컴퓨터에서 `node server.js`로 그냥 실행할 때는 `PORT`
환경변수가 없으므로 지금까지와 똑같이 3000번 포트로 실행됩니다(직접 실행해서
`http://localhost:3000`으로 확인했고, `PORT` 값을 넣었을 때도 그 포트로 정상 실행되는
것까지 확인했습니다).

### Render 배포 설정값 (초보자용)

[render.com](https://render.com)에서 "New +" → "Web Service"로 새 서비스를 만들 때
아래 값을 그대로 입력하면 됩니다.

| 설정 항목 | 입력할 값 |
| --- | --- |
| Service Type | **Web Service** |
| Language(Runtime) | **Node** |
| Branch | **main** |
| Build Command | `npm install` (또는 비워둬도 됩니다 — 이 프로젝트는 외부 패키지를 쓰지 않습니다) |
| Start Command | `node server.js` |

**환경변수(Environment Variables)** 설정 화면에 아래 4개 이름을 만들고, 각각 실제
네이버 API에서 발급받은 값을 입력합니다.

```
NAVER_SHOPPING_CLIENT_ID
NAVER_SHOPPING_CLIENT_SECRET
NAVER_SEARCH_TREND_CLIENT_ID
NAVER_SEARCH_TREND_CLIENT_SECRET
```

(용도별 키 대신 공용 키 한 쌍만 있다면 `NAVER_CLIENT_ID`, `NAVER_CLIENT_SECRET` 두
개만 입력해도 됩니다. `.env.example`에 설명된 규칙과 동일합니다.)

> ⚠️ **중요**: 실제 API 키/Secret 값은 **GitHub 저장소(코드)나 `.env` 파일에 커밋해서
> 올리는 것이 아니라, 반드시 Render의 "Environment Variables" 설정 화면에 직접
> 입력**해야 합니다. GitHub에는 `.env.example`처럼 값이 비어있는 안내 파일만 올라가야
> 하고, `.gitignore`가 `.env` 파일 자체를 Git에서 제외해줍니다.

## v2.1 — Render 배포 화면 표시 문제 수정

Render에 배포한 주소(`https://trend-top10.onrender.com`)로 접속했을 때 "Not Found"가
뜨는 문제를 고쳤습니다. 원인은 `server.js`가 `/api/...` 같은 API 주소만 처리할 뿐,
메인 화면 주소(`/`)로 접속했을 때 `index.html`을 보여주는 코드가 없었기 때문입니다.
API 키/Secret은 이번에도 `index.html`에 넣지 않았고, `.env` 파일도 건드리지
않았습니다.

### 실제로 바뀐 부분

**1) `server.js`: `GET /` 요청에 `index.html`을 응답하도록 추가**

```js
if (pathname === "/") {
  var indexPath = path.join(__dirname, "index.html");
  fs.readFile(indexPath, "utf8", function (err, html) {
    // 성공하면 index.html 내용을 text/html로 그대로 돌려줌
  });
  return;
}
```

이제 누군가 서버 주소(`/`)로 접속하면, 서버가 같은 폴더에 있는 `index.html` 파일을
읽어서 그대로 화면에 보여줍니다. 기존 API 주소 4개(`/api/trends`, `/api/naver-ready`,
`/api/search-trend-test`, `/api/shopping-insight-test`)는 이전과 완전히 동일하게
동작합니다.

**2) `index.html`: API 서버 주소를 여는 방식에 맞게 자동으로 선택**

이전에는 `http://localhost:3000/api/trends` 주소가 코드에 그대로 고정되어 있어서,
Render에 배포된 화면에서는 항상 내 컴퓨터의 localhost를 찾다가 실패했습니다. 이제는
아래처럼 지금 이 페이지가 어떻게 열렸는지 확인해서 주소를 자동으로 고릅니다.

```js
function getApiBaseUrl() {
  if (window.location.protocol === "file:") {
    // 컴퓨터에서 index.html 파일을 더블클릭해서 열었을 때
    return "http://localhost:3000/api/trends";
  }
  // Render처럼 웹 서버가 index.html까지 함께 내려주는 경우
  return "/api/trends";
}
```

- **로컬에서 `index.html`을 더블클릭해서 열 때**(주소창에 `file://...`로 시작):
  지금까지와 똑같이 `http://localhost:3000/api/trends`로 요청합니다. `node server.js`를
  먼저 실행해둬야 하는 것도 그대로입니다.
- **Render 주소(`https://...`)로 접속했을 때**: `/api/trends`라는 "상대 주소"로
  요청합니다. 상대 주소는 지금 보고 있는 페이지와 같은 서버로 자동 연결되므로,
  Render 주소를 코드에 직접 적을 필요가 없습니다.

### 브라우저에서 확인할 항목 (초보자용)

배포 후 아래 3개 주소를 각각 열어서 확인해 주세요.

1. **`https://trend-top10.onrender.com`**
   "Not Found" 대신 "트렌드 TOP 10" 화면(제목, 키워드 입력 영역 등)이 정상적으로
   보이는지 확인합니다. 페이지가 열리자마자 자동으로 트렌드 조회가 실행되어 표/차트에
   데이터가 채워지는지도 함께 확인합니다.
2. **`https://trend-top10.onrender.com/api/naver-ready`**
   `{"shoppingInsightReady":true,"searchTrendReady":true}`처럼 JSON 응답이 그대로
   보이는지 확인합니다(이전과 동일하게 동작해야 합니다).
3. **`https://trend-top10.onrender.com/api/trends`**
   `searchData`, `shoppingProductData` 등이 담긴 JSON 응답이 보이는지 확인합니다
   (이전과 동일하게 동작해야 합니다).

로컬 확인도 잊지 마세요: `node server.js`를 실행한 뒤 `index.html`을 더블클릭해서
열었을 때도 이전과 동일하게 잘 동작해야 합니다(이번 변경으로 로컬 동작 방식은
바뀌지 않았습니다).

### v2.1 백업 생성

Render 배포에서 정상 작동하는 v2.1 시점을 스냅샷으로 남겨두기 위해
`backups/index_v2.1.html`과 `backups/server_v2.1.js`를 새로 만들었습니다. 두 파일 모두
지금 `index.html` / `server.js`와 내용이 동일한 "그대로 복사본"이며, 나중에 기능을
수정하다가 문제가 생기면 이 시점(Render에서 실제로 정상 동작을 확인한 시점)으로
되돌아올 수 있습니다. v2.0~v2.1 사이에 `server.js`(포트 처리, `/` 라우트 추가)와
`index.html`(API 주소 자동 분기)이 모두 바뀌었기 때문에, 이번에는 두 파일 모두
새로 백업했습니다.
